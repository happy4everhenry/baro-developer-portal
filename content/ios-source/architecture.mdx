## MVVM Pattern Implementation

### View Layer

Views are pure SwiftUI structs that declare UI and bind to ViewModels via `@State`:

```swift
struct HomeView: View {
    @State private var viewModel: HomeViewModel

    init(container: DependencyContainer) {
        _viewModel = State(initialValue: HomeViewModel(
            marketRepository: container.marketRepository,
            bannerRepository: container.bannerRepository,
            newsRepository: container.newsRepository
        ))
    }

    var body: some View {
        ScrollView {
            // Sections bind directly to viewModel properties
            StockListView(stocks: viewModel.stocks,
                         latestPrices: viewModel.latestPrices)
        }
        .task { await viewModel.loadAllData() }
    }
}
```

**Key patterns:**
- Views hold `@State var viewModel` — the modern iOS 17+ approach
- Data loading is triggered via `.task { }` modifier
- Views never call repositories directly — always through the ViewModel
- Navigation uses `NavigationLink` and `NavigationStack`

### ViewModel Layer

ViewModels are `@Observable` classes that manage state and coordinate data fetching:

```swift
@Observable
final class HomeViewModel {
    // MARK: - Dependencies
    private let marketRepository: any MarketRepositoryProtocol
    private let bannerRepository: any BannerRepositoryProtocol

    // MARK: - Published State
    var stocks: [Stock] = []
    var indices: [MarketIndex] = []
    var isLoadingStocks = true
    var latestPrices: [String: PriceUpdate] = [:]

    // MARK: - Data Loading
    func loadAllData() async {
        async let indicesTask = loadIndices()
        async let stocksTask = loadStocks()
        async let bannersTask = loadBanners()
        _ = await (indicesTask, stocksTask, bannersTask)
    }
}
```

**Key patterns:**
- `@Observable` macro (not legacy `ObservableObject`) — automatic change tracking
- Per-section loading states (`isLoadingStocks`, `isLoadingBanners`, etc.)
- Parallel data loading with `async let`
- WebSocket integration via `AsyncStream` for live price updates
- Filter/exchange selection triggers automatic reload

### Repository Layer

Repositories abstract data access behind protocols, enabling mock/API swapping:

```swift
protocol MarketRepositoryProtocol: Sendable {
    func fetchStocks(filter: StockFilter, exchange: StockExchange) async throws -> [Stock]
    func fetchStockDetail(id: String, exchange: StockExchange,
                          name: String, venue: MarketVenue) async throws -> StockDetailInfo
    // ... 9 methods total
}
```

Two implementations:
- **`APIMarketRepository`** — calls the FastAPI backend via `APIClient`, falls back to mock on failure
- **`MockMarketRepository`** — generates deterministic mock data with seeded randomization

## Dependency Injection

All repositories are created and held by `DependencyContainer`, which is injected into the SwiftUI environment:

```swift
// BAROApp.swift
@main
struct BAROApp: App {
    @State private var container = DependencyContainer(useMocks: true)

    var body: some Scene {
        WindowGroup {
            MainTabView()
                .environment(container)
        }
    }
}
```

```swift
// DependencyContainer.swift
@Observable
final class DependencyContainer {
    let marketRepository: any MarketRepositoryProtocol
    let bannerRepository: any BannerRepositoryProtocol
    let newsRepository: any NewsRepositoryProtocol

    init(useMocks: Bool = false) {
        if useMocks {
            marketRepository = MockMarketRepository()
            // ...
        } else {
            marketRepository = APIMarketRepository()
            // ...
        }
    }
}
```

**Switching between mock and live data** is a single boolean toggle in `BAROApp.swift`.

## Key Architectural Decisions

### Why @Observable over ObservableObject

The app uses Swift's modern `@Observable` macro (Observation framework) instead of the legacy `ObservableObject` + `@Published` pattern:

| Aspect | @Observable (Used) | ObservableObject (Legacy) |
|--------|-------------------|--------------------------|
| Change tracking | Automatic, per-property | Manual via `@Published` |
| View invalidation | Only properties actually read | All `@Published` properties |
| Boilerplate | None — just add `@Observable` | `@Published` on every property |
| Performance | Fine-grained updates | Potentially over-renders |

### Why Protocol-Based Repositories

- **Testability** — swap `MockMarketRepository` in tests without touching ViewModels
- **Development speed** — UI development proceeds with mock data while API is under construction
- **Resilience** — `APIMarketRepository` falls back to mock on network failure
- **Separation of concerns** — ViewModels don't know how data is fetched

### Why Singleton WebSocketManager

- **Connection pooling** — single WebSocket connection shared across all screens
- **Subscription management** — subscribe/unsubscribe by stock code, avoiding duplicate connections
- **Lifecycle management** — automatic reconnection with exponential backoff
- **Venue awareness** — subscription messages include venue (KRX/NXT/Combined)
