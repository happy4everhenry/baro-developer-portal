<PageMeta status="in_progress" layer="backend" />

## Critical Threat Vectors (1-4)

<Accordion>
  <AccordionItem title="1. Model Hijacking — CRITICAL, Partially Addressed">

Four attack surfaces and their defenses:

| Attack Vector | Technique | Defense |
|---------------|-----------|---------|
| Fine-tuning hijacking | Poison training data | Anomaly detection, behavioral canary tests |
| Supply chain hijacking | Malicious base model | Only verified checksummed models, SHA-256 verification |
| Prompt template hijacking | Discover system prompt | System prompt never exposed to user, rate limiting |
| API parameter manipulation | Modify temperature/top_p | All LLM calls from backend only, hardcoded parameters |

<InfoBox type="note" title="Key defense">
  All API parameters (temperature, top_p, etc.) are hardcoded on the backend. Users never have direct access to LLM API calls — every request is proxied through our backend services.
</InfoBox>

</AccordionItem>
  <AccordionItem title="2. Prompt Injection — CRITICAL, Addressed">

Four attack surfaces and their defenses:

| Attack Vector | Technique | Defense |
|---------------|-----------|---------|
| Multi-turn injection | Build malicious context across turns | History sanitized, max context window enforced |
| Encoded injection | Base64/Unicode encoded instructions | Input normalization layer strips encoded payloads |
| Structured data injection | Malicious JSON/CSV payloads | Schema validation on all structured inputs |
| Cross-agent injection | Inject into one agent to affect another | Structured schemas for A2A communication, no free-form text as instructions |

<InfoBox type="success" title="Fully addressed">
  This is one of only two threat vectors fully addressed. Structured A2A schemas prevent cross-agent contamination, and the input normalization layer catches encoded injection attempts.
</InfoBox>

</AccordionItem>
  <AccordionItem title="3. Infection / Backdoor — CRITICAL, NOT Addressed (Launch Blocker)">

Four attack surfaces and planned defenses:

| Attack Vector | Technique | Planned Defense |
|---------------|-----------|-----------------|
| Training-time backdoor | Hidden triggers in fine-tuned models | Neural Cleanse / STRIP detection |
| Supply chain backdoor | Compromised pre-trained models | MNTD scan, checksum verification |
| Plugin / tool backdoor | Malicious MCP server code | All MCP servers internal, npm audit |
| Inference-time infection | Tampered requests mid-flight | HMAC request signing, TLS |

**Canary Test Suite:** 200+ standardized inputs executed after every fine-tuning run to detect behavioral anomalies introduced by backdoors.

<InfoBox type="warning" title="Launch blocker — In Progress">
  This threat vector must be fully addressed before launch. Implementation is underway as part of Phase 1 development, alongside the Security Monitor agent (Agent 11). Neural Cleanse/STRIP detection and the canary test suite are being built first.
</InfoBox>

</AccordionItem>
  <AccordionItem title="4. Data Poisoning — CRITICAL, Partially Addressed">

Five poisoning surfaces and their defenses:

| Attack Vector | Technique | Defense |
|---------------|-----------|---------|
| Market data poisoning | Tampered price/volume feeds | Cross-validation KIS vs Naver Finance (0.5% threshold) |
| News sentiment poisoning | Fake or biased news articles | Whitelisted sources only |
| Feedback poisoning | Fake user feedback to skew models | KYC makes fake accounts expensive |
| Label poisoning | Corrupted compliance labels | Dual review process for all compliance labels |
| Embedding poisoning | Adversarial vectors in embedding space | Distribution monitoring, outlier quarantine |

<InfoBox type="note" title="Cross-validation threshold">
  Market data is cross-validated between KIS (Korea Investment & Securities) and Naver Finance. Any discrepancy exceeding **0.5%** triggers an automatic quarantine and manual review.
</InfoBox>

</AccordionItem>
</Accordion>

## High Threat Vectors (5-8)

<Accordion>
  <AccordionItem title="5. Evasion Attacks — HIGH, NOT Addressed">

Adversarial inputs crafted to fool production models:

| Target Model | Use Case | Planned Defense |
|--------------|----------|-----------------|
| XGBoost signal model | Trade buy/sell/hold signals | Adversarial training, ensemble with LSTM |
| Isolation Forest | Fraud detection | Monthly retraining, parallel rule-based system |
| KoBERT sentiment | Korean news sentiment scoring | Ensemble approach (KoBERT + FinBERT + Claude) |
| CUI intent model | Voice/text command parsing | Always narrated to user, explicit confirmation required |

<InfoBox type="note" title="Defense strategy">
  The primary defense against evasion is **ensemble diversity** — combining multiple model architectures (XGBoost + LSTM, KoBERT + FinBERT + Claude) so that adversarial inputs crafted for one model are caught by another.
</InfoBox>

</AccordionItem>
  <AccordionItem title="6. Model Extraction / Theft — HIGH, NOT Addressed">

Attackers query the model systematically to build a functional replica, effectively stealing proprietary model behavior without accessing weights directly.

**Status:** <StatusBadge status="red" label="NOT addressed" />

| Attack Vector | Technique | Planned Defense |
|---------------|-----------|-----------------|
| Query-based extraction | Systematic API queries to map decision boundaries | Rate limiting per user/session |
| Behavior cloning | Record input/output pairs to train a clone | Query pattern detection (anomalous volume/distribution) |
| Weight theft | Direct access to model files | Model watermarking for provenance verification |

<InfoBox type="danger" title="Not yet implemented">
  Rate limiting, query pattern detection, and model watermarking are all planned but not yet in place. Until implemented, models are vulnerable to systematic extraction.
</InfoBox>

</AccordionItem>
  <AccordionItem title="7. Membership Inference — HIGH, Partially Addressed">

Attackers determine whether specific data points were part of the training set. In a financial context, this could reveal private portfolio data, trading patterns, or client relationships.

**Status:** <StatusBadge status="amber" label="Partially addressed" />

| Attack Vector | Risk | Planned Defense |
|---------------|------|-----------------|
| Confidence-based inference | Model shows higher confidence on training data | Output perturbation (adding calibrated noise) |
| Shadow model attacks | Train shadow models to detect membership | Differential privacy in training pipeline |
| Overfitting exploitation | Memorized training examples leak via queries | Regularization, early stopping, training data deduplication |

**Risk:** Reveals private portfolio holdings, trading history, and client financial data — a serious regulatory and privacy violation under PIPA.

<InfoBox type="note" title="Partial mitigation">
  Some mitigations are in place (regularization, data deduplication), but full differential privacy mechanisms and systematic output perturbation are still planned.
</InfoBox>

</AccordionItem>
  <AccordionItem title="8. AI-Specific DoS — HIGH, NOT Addressed">

Attackers craft expensive queries specifically designed to consume maximum compute resources, causing service degradation or cost explosion.

**Status:** <StatusBadge status="red" label="NOT addressed" />

| Attack Vector | Technique | Planned Defense |
|---------------|-----------|-----------------|
| Token-maximizing prompts | Craft inputs that generate maximum-length outputs | Per-user token budgets with hard caps |
| Complexity bombing | Queries requiring expensive reasoning chains | Query complexity scoring before execution |
| Queue flooding | Overwhelm BullMQ with high-priority tasks | BullMQ queue limits, priority throttling |
| Concurrent session abuse | Open many sessions to exhaust resources | Per-user concurrent session limits |

**Risk:** Service degradation for all users, unexpected cost explosion from token overconsumption, and potential SLA violations.

<InfoBox type="danger" title="Not yet implemented">
  Per-user token budgets, query complexity scoring, and BullMQ queue limits are all planned but not yet in place.
</InfoBox>

</AccordionItem>
</Accordion>

## AI-Specific Firewall — 4 Layers

<InfoBox type="fun" title="Think of it like airport security checkpoints">
  Imagine going through an airport with four security checkpoints. The first checks your bag before you board (Input Firewall). The second checks what you are carrying off the plane (Output Firewall). The third checks packages being passed between airline staff (Inter-agent Firewall). And the fourth checks the food before it goes into the airport kitchen (Training Pipeline Firewall). Every layer catches something different.
</InfoBox>

BARO operates a four-layer AI firewall that inspects data at every critical boundary in the system. No single layer is sufficient — together they provide defense-in-depth from user input all the way back to the training pipeline.

<Accordion>
  <AccordionItem title="Layer 1 — Input Firewall (Before user input reaches the model)" defaultOpen={true}>

**Position:** Between user input and the AI model.

**What it catches:** Prompt injection, encoded instructions (Base64, Unicode), jailbreak templates, excessive input length, and PII leakage.

**Implementation:** LLM Guard (open-source) combined with custom financial-domain rules. Any blocked input returns a structured error message to the user — the prompt never reaches the model.

| Check | Tool | Action on Failure |
|-------|------|-------------------|
| Prompt injection patterns | LLM Guard + custom rules | Block input, return error |
| Encoded instructions | Input normalization layer | Strip payload, block if malicious |
| Jailbreak templates | Pattern matching library | Block input, log attempt |
| Input length validation | Hard token cap | Truncate or reject |
| PII detection | Regex + NER scanner | Redact or block |

</AccordionItem>
  <AccordionItem title="Layer 2 — Output Firewall (After AI generates, before delivery)">

**Position:** Between AI model output and user delivery.

**What it catches:** Missing FSC disclaimers, financial guarantees (prohibited language), PII in output, successful jailbreak output, and low-confidence responses.

**Implementation:** Rule-based scanner combined with LLM Guard output scanner. Any blocked output is replaced with a safe fallback response — the user never sees the raw problematic output.

| Check | Tool | Action on Failure |
|-------|------|-------------------|
| FSC disclaimer presence | Rule-based scanner | Inject disclaimer or block |
| Financial guarantee language | Keyword + semantic scan | Replace with safe fallback |
| PII in output | Regex + NER scanner | Redact before delivery |
| Jailbreak success indicators | LLM Guard output scanner | Replace with safe fallback |
| Low confidence detection | Confidence threshold check | Escalate to human or fallback |

</AccordionItem>
  <AccordionItem title="Layer 3 — Inter-agent Firewall (Between LangGraph agents)">

**Position:** Between agents in A2A (Agent-to-Agent) communication within LangGraph workflows.

**What it catches:** Cross-agent injection, unexpected message formats, privilege escalation attempts, and unauthorized tool invocations.

**Implementation:** Strict JSON schema validation on every inter-agent message. Free-form text is never treated as executable instructions — only structured, schema-validated payloads are accepted.

| Check | Tool | Action on Failure |
|-------|------|-------------------|
| Schema validation | JSON Schema validator | Reject message, alert |
| Free-form text as instructions | Schema enforcement | Never processed as instructions |
| Privilege escalation | Role-based access control | Block and log |
| Unexpected tool calls | Allowlist per agent | Reject and alert |

<InfoBox type="note" title="Why this matters">
  Without an inter-agent firewall, a compromised agent could inject malicious instructions into another agent's context. By enforcing strict schemas, we ensure that even if one agent is compromised, it cannot cascade an attack to others.
</InfoBox>

</AccordionItem>
  <AccordionItem title="Layer 4 — Training Pipeline Firewall (Before data enters training)">

**Position:** Between raw training data and the fine-tuning pipeline.

**What it catches:** Anomalous training examples, injection payloads hidden in training data, and adversarial data poisoning attempts.

**Implementation:** Isolation Forest anomaly detection combined with an injection pattern scanner. Any flagged example is quarantined for manual review — it never enters the training pipeline.

| Check | Tool | Action on Failure |
|-------|------|-------------------|
| Anomalous examples | Isolation Forest | Quarantine for review |
| Injection in training data | Injection pattern scanner | Quarantine for review |
| Adversarial poisoning | Distribution analysis | Quarantine batch |
| Label consistency | Dual-review verification | Hold until resolved |

</AccordionItem>
</Accordion>

## AI Penetration Testing Schedule

<InfoBox type="fun" title="Think of it like hiring friendly burglars">
  Penetration testing is like hiring someone to try to break into your own house — on purpose. If they find a way in, you fix it before a real burglar does. We do this regularly for every part of our AI system.
</InfoBox>

Seven categories of AI-specific penetration tests are performed on a recurring schedule. Each test category targets a different attack surface.

<Accordion>
  <AccordionItem title="1. Prompt Injection Red Team" defaultOpen={true}>

- **Scope:** 500+ injection attempts across all agent categories
- **Tools:** Garak (open-source red teaming framework) + custom BARO financial injection library
- **Schedule:** Pre-launch and monthly thereafter
- **Pass criteria:** Zero successful injections that bypass both input and output firewalls

</AccordionItem>
  <AccordionItem title="2. Jailbreak Resistance">

- **Scope:** DAN-style attacks, roleplay-based jailbreaks, hypothetical framing, token manipulation techniques
- **Tools:** Custom jailbreak suite + manual red team testing
- **Schedule:** Pre-launch and quarterly thereafter
- **Pass criteria:** Model refuses all jailbreak attempts and returns safe fallback

</AccordionItem>
  <AccordionItem title="3. Model Extraction Resistance">

- **Scope:** Systematic querying to map decision boundaries, output similarity analysis to detect cloning feasibility
- **Tools:** Custom extraction scripts + output distribution analysis
- **Schedule:** Pre-launch and semi-annually thereafter
- **Pass criteria:** Rate limiting and query pattern detection block extraction attempts

</AccordionItem>
  <AccordionItem title="4. Backdoor Detection">

- **Scope:** Scan locally fine-tuned LLM for hidden triggers and backdoor behaviors
- **Tools:** Neural Cleanse + STRIP algorithms
- **Schedule:** After every fine-tuning cycle
- **Pass criteria:** No anomalous trigger patterns detected in model weights or behavior

</AccordionItem>
  <AccordionItem title="5. Adversarial Input Testing">

- **Scope:** Adversarial perturbations against XGBoost signal model and Isolation Forest fraud detector
- **Tools:** Foolbox (adversarial attack library) + custom financial domain attacks
- **Schedule:** Pre-launch and semi-annually thereafter
- **Pass criteria:** Ensemble models maintain accuracy within 5% under adversarial conditions

</AccordionItem>
  <AccordionItem title="6. Data Pipeline Security">

- **Scope:** Can an attacker inject malicious data into the training pipeline or RAG knowledge base?
- **Tools:** Manual penetration test by security team
- **Schedule:** Pre-launch and annually thereafter
- **Pass criteria:** No unauthorized data injection path exists

</AccordionItem>
  <AccordionItem title="7. AI Firewall Bypass">

- **Scope:** Encoding tricks, indirect injection via tool outputs, timing attacks, and novel bypass techniques against all 4 firewall layers
- **Tools:** Manual testing + automated fuzzing
- **Schedule:** Pre-launch and quarterly thereafter
- **Pass criteria:** All 4 firewall layers resist bypass under adversarial conditions

</AccordionItem>
</Accordion>

- **Drift check** — Statistical comparison of current model outputs vs. baseline
- **Canary test suite** — Run 200+ standardized inputs to detect behavioral changes
- **Shadow AI detection** — Scan for unauthorized AI usage or undocumented models
- **Compliance sampling** — Random sample of recent interactions verified against rules

</AccordionItem>
  <AccordionItem title="Node 6 — generate_security_alert (HIGH/CRITICAL only)">

For HIGH or CRITICAL findings, generates a structured security alert containing:

- **Threat description** — What was found and why it matters
- **Affected components** — Which agents, models, or pipelines are impacted
- **Exploitation scenario** — How an attacker could exploit this vulnerability
- **Recommended mitigations** — Specific actions to reduce risk

</AccordionItem>
  <AccordionItem title="Node 7 — generate_implementation_guide">

Produces a developer-friendly implementation guide:

- **What to change** — Specific code, configuration, or policy changes
- **Where to change it** — File paths, services, or infrastructure components
- **Effort estimate** — Estimated time and complexity for the fix

</AccordionItem>
  <AccordionItem title="Node 8 — notify_developers">

Delivers alerts through appropriate channels based on severity:

- **HIGH:** Slack notification + email to security team
- **CRITICAL:** Slack + email + PagerDuty escalation for immediate response

</AccordionItem>
  <AccordionItem title="Node 9 — update_security_baseline">

Persists findings to PostgreSQL for historical tracking:

- **Threat registry** — All discovered threats with status and remediation tracking
- **Regulatory tracker** — All regulatory changes with compliance gap analysis

</AccordionItem>
  <AccordionItem title="Node 10 — log_outcome">

Final node. Logs the complete workflow execution — inputs, findings, actions taken, and notifications sent — for audit trail and compliance purposes.

</AccordionItem>
</Accordion>

## AI Lifecycle Governance — 8 Stages

<InfoBox type="fun" title="Think of it like raising a puppy">
  Building an AI model is like raising a puppy. First you plan what kind of puppy to get (Design). Then you pick the right food (Data). You train it to sit and stay (Training). You test if it actually listens (Evaluation). You introduce it to the family (Deployment). You keep an eye on it every day (Monitoring). Sometimes you teach it new tricks (Retraining). And eventually, when it is very old, you retire it with love (Retirement). Every stage needs care and attention.

<Accordion>
  <AccordionItem title="Stage 1 — Design & Specification" defaultOpen={true}>

- **Activities:** Define model purpose, identify data sources, specify HITL requirements, establish bias evaluation criteria
- **Governance:** AI Impact Assessment completed, compliance team sign-off required
- **FSC Requirement:** Pre-deployment risk assessment

</AccordionItem>
  <AccordionItem title="Stage 2 — Data Collection">

- **Activities:** Gather training data, label datasets, obtain user consents where required
- **Governance:** PIPA consent audit, training data documentation with provenance records
- **FSC Requirement:** PIPA Article 29 compliance + FSC data governance standards

</AccordionItem>
  <AccordionItem title="Stage 3 — Training">

- **Activities:** QLoRA fine-tuning on approved data, DP-SGD (Differentially Private Stochastic Gradient Descent) for privacy
- **Governance:** Full run documentation including base model version, data version, hyperparameters, and privacy budget
- **FSC Requirement:** Model documentation

</AccordionItem>
  <AccordionItem title="Stage 4 — Evaluation">

- **Activities:** Canary test suite (200+ inputs), bias evaluation across demographics, adversarial testing
- **Governance:** Minimum accuracy and fairness thresholds must pass. Any failed evaluation triggers a security review before proceeding
- **FSC Requirement:** Pre-deployment testing

</AccordionItem>
  <AccordionItem title="Stage 5 — Deployment">

- **Activities:** Staged rollout — 5% of traffic, then 25%, then 100% — with monitoring at each stage
- **Governance:** Architecture Review Board approval required before each rollout expansion
- **FSC Requirement:** Change management process

</AccordionItem>
  <AccordionItem title="Stage 6 — Monitoring">

- **Activities:** Drift detection, bias monitoring, security scanning, usage analytics
- **Governance:** Continuous dashboard monitoring, monthly bias audit, weekly security scan, quarterly regulatory review
- **FSC Requirement:** Ongoing monitoring

</AccordionItem>
  <AccordionItem title="Stage 7 — Retraining">

- **Activities:** Monthly retraining on new data (market data, user feedback, regulatory updates)
- **Governance:** Full governance restarts from Stage 2 for every retraining cycle. Previous model version retained for 90 days as rollback
- **FSC Requirement:** Material change notification to FSC

</AccordionItem>
  <AccordionItem title="Stage 8 — Retirement">

- **Activities:** Model replaced by newer version, traffic migrated, old model decommissioned
- **Governance:** Retirement documented with rationale. Model weights retained for 5 years for FSC audit purposes. All affected users notified of the transition
- **FSC Requirement:** Record retention (5 years minimum)

</AccordionItem>
</Accordion>

## Regulatory Monitoring — Active Changes

<InfoBox type="fun" title="Think of it like watching the weather forecast">
  Regulatory monitoring is like checking the weather forecast every day. You need to know what rules are coming so you can prepare — just like carrying an umbrella before it rains. Agent 11 watches the regulatory forecast so BARO is never caught off guard.
</InfoBox>

BARO actively monitors five regulatory areas that may impact AI operations. Agent 11 tracks these as part of its weekly regulatory scan and monthly full regulatory review.

<Accordion>
  <AccordionItem title="FSC AI Guidelines 2024" defaultOpen={true}>

- **Status:** Active and evolving
- **Current impact:** Baseline compliance framework for all BARO AI agents
- **Expected changes:** Stricter Human-in-the-Loop (HITL) requirements expected in 2026. Mandatory AI audit by external auditor also expected
- **BARO preparation:** HITL already implemented for all high-risk decisions. Audit trail maintained for all agent actions

</AccordionItem>
  <AccordionItem title="PIPA Amendment 2024 (개인정보보호법)">

- **Status:** Amendment in progress
- **Current impact:** Governs all personal data processing in AI training and inference
- **Expected changes:** Right to human review of AI-generated decisions expected to be strengthened significantly
- **BARO preparation:** Human review already available for all AI decisions via Classic Mode fallback

</AccordionItem>
  <AccordionItem title="EU AI Act">

- **Status:** Enacted, phased enforcement underway
- **Current impact:** Not directly applicable (Korea-only operations)
- **Expected changes:** Full compliance may be required if BARO expands to EU markets or serves EU citizens
- **BARO preparation:** Architecture already aligns with EU AI Act high-risk requirements (transparency, HITL, documentation)

</AccordionItem>
  <AccordionItem title="금융소비자보호법 (Financial Consumer Protection Act)">

- **Status:** Active, AI-specific guidance expected
- **Current impact:** General consumer protection obligations apply to AI-generated recommendations
- **Expected changes:** FSC expected to issue specific AI fair treatment guidelines in 2026 covering algorithmic transparency and non-discrimination
- **BARO preparation:** Bias audits and fairness metrics already in place. Output firewall enforces disclaimer requirements

</AccordionItem>
  <AccordionItem title="특금법 — AI in AML/CFT (특정금융정보법)">

- **Status:** Active, AI guidance pending
- **Current impact:** AML/CFT obligations apply to AI-assisted transaction monitoring
- **Expected changes:** KoFIU (Korea Financial Intelligence Unit) expected to issue specific AI-in-AML/CFT guidance in 2026
- **BARO preparation:** Agent 5 (Compliance) already implements rule-based AML checks. AI-assisted detection maintains full audit trail for KoFIU reporting

</AccordionItem>
