  <AccordionItem title="Layer 1 — Infrastructure" icon="🏗️" badge="Foundation" defaultOpen={true}>

**What it does:** Provides compute, storage, networking, and defines latency tiers for the entire platform.

**Key Technologies:**
- **Vercel** — Frontend hosting & edge functions
- **Railway Seoul** — Backend containers (Korea region)
- **Supabase** — Managed PostgreSQL + Auth
- **Upstash Redis** — Serverless Redis for caching & pub/sub
- **Cloudflare** — CDN, DDoS protection, edge workers

**3 Latency Tiers:**

| Tier | Target | Example Use Cases |
|------|--------|-------------------|
| **Critical** | under 100ms | Auth checks, cache reads, order validation |
| **Interactive** | 100ms – 1s | Chat responses, search, chart rendering |
| **Background** | 1s – 60s | Report generation, model training, batch jobs |

<InfoBox type="tip" title="Kid-friendly analogy">
  Think of latency tiers like traffic lanes on a highway. The fast lane (Critical) is for ambulances — things that absolutely cannot wait. The middle lane (Interactive) is for regular cars — fast but not urgent. The slow lane (Background) is for big trucks carrying heavy loads — they take longer but carry more stuff.
</InfoBox>

</AccordionItem>

  <AccordionItem title="Layer 2 — Model" icon="🧠" badge="AI Brains">

**What it does:** Houses all LLMs, ML classifiers, and embedding models used by the agents.

**3 Model Categories:**

| Category | Purpose | Models |
|----------|---------|--------|
| **A: Cloud LLM** | Complex reasoning, natural language | Claude API (Anthropic) |
| **B: Local ML** | Classification, anomaly detection, NLP | XGBoost, Isolation Forest, LSTM, KoBERT, FinBERT-KO |
| **C: Embeddings** | Vector search, semantic similarity | BGE-M3 (primary), text-embedding-3-small (fallback) |

**Additional models:**
- **EXAONE 3.5** — Korean-optimized LLM for local inference

<InfoBox type="note" title="Why three categories?">
  Category A (Claude) is like asking a brilliant professor a question — powerful but costs money per question. Category B is like having a fast calculator on your desk — it runs locally and is free per query. Category C turns words into numbers so we can find similar things quickly.
</InfoBox>

</AccordionItem>

  <AccordionItem title="Layer 3 — Data" icon="💾" badge="Knowledge">

**What it does:** Ingests, stores, and serves all market data, user data, and vector embeddings.

**Data Sources:**

| Source | Data Type |
|--------|-----------|
| **KIS Open API** | Korean stock market (실시간 시세) |
| **Upbit + Binance** | Cryptocurrency prices |
| **OANDA** | Forex rates |
| **Naver Finance + RSS** | Korean financial news |
| **User Events** | Clicks, trades, preferences |
| **PostgreSQL** | Relational application data |
| **오픈뱅킹 API** | Open banking integration |

**Storage & Processing:**
- **TimescaleDB** — Time-series market data
- **pgvector** — Vector store for RAG/embeddings
- **Redis** — Real-time cache & pub/sub
- **TA-Lib** — Technical analysis calculations (SMA, RSI, MACD, etc.)

</AccordionItem>

  <AccordionItem title="Layer 4 — Orchestration" icon="🔄" badge="Traffic Control">

**What it does:** Manages agent workflows, human-in-the-loop (HITL) approvals, and agent-to-agent (A2A) events.

**Key Technologies:**
- **LangGraph** — Typed state graphs with conditional edges
- **BullMQ** — Persistent job queues for background tasks
- **Redis Pub/Sub** — Real-time event broadcasting between agents

<InfoBox type="fun" title="Like an air traffic controller">
  The orchestration layer is like an air traffic controller at a busy airport. It decides which agent (airplane) goes next, makes sure they don't crash into each other, and always asks a human for permission before anything risky happens (like landing on a new runway).
</InfoBox>

</AccordionItem>

  <AccordionItem title="Layer 5 — Application" icon="📱" badge="User-Facing">

**What it does:** Delivers the UI to users, streams AI responses in real time, and renders charts.

**Key Technologies:**
- **Next.js 14** — Web app with App Router & Server Components
- **React Native** — Mobile app (iOS & Android)
- **TradingView Charts** — Interactive financial charting library

</AccordionItem>

  <AccordionItem title="Layer 6 — MCP" icon="🔌" badge="Tool Interface">

**What it does:** Provides 6 Model Context Protocol servers that give AI agents structured access to external tools and data.

MCP (Model Context Protocol) is an open standard by Anthropic that lets AI models call tools in a safe, structured way — like giving each robot a specific toolbox.

<InfoBox type="info" title="What is MCP?">
  Imagine each AI agent has a walkie-talkie. MCP is the radio frequency they all share — it lets them call specific tools (like "fetch stock price" or "search news") in a way that's safe and standardized. Details on each of the 6 MCP servers live in a separate document.
</InfoBox>

</AccordionItem>

  <AccordionItem title="Layer 7 — LangGraph" icon="🧩" badge="State Machine">

**What it does:** Defines typed state graphs with conditional edges, enabling complex multi-step agent workflows with full checkpointing.

**Key Capabilities:**
- **Typed State Graphs** — Each agent's workflow is a graph with strongly-typed state at every node
- **Conditional Edges** — "If the risk score is above 0.8, route to human review; otherwise, proceed"
- **Checkpointing** — Full agent state persisted to Redis at every node transition

**Checkpoint & Recovery:**
- Every node transition saves the full state to Redis
- HITL pause states are checkpointed
- If the server restarts, the agent resumes **exactly** where it paused — no lost progress

<InfoBox type="tip" title="Like a video game save point">
  LangGraph checkpointing works just like save points in a video game. Every time an agent finishes a step, it saves its progress. If the game (server) crashes, you don't start over from the beginning — you reload your last save and keep going.
</InfoBox>

</AccordionItem>

## Full Agent Workflows

<InfoBox type="info" title="What is in each workflow?">
  Every agent has two things: a **state** (the data it carries around, like a clipboard) and a **workflow** (the step-by-step recipe it follows). Below you will find both for all 12 agents. Think of the state like a robot's memory, and the workflow like its instruction manual.
</InfoBox>

<InfoBox type="warning" title="FSC Hard Rule: Human-in-the-Loop">
  Nodes marked **HITL** require a human to approve before the workflow can continue. This is a Korean Financial Services Commission (FSC) requirement — no exceptions, even in "Auto Mode." If the timeout expires without a decision, the workflow either cancels or escalates.
</InfoBox>

<Accordion>
  <AccordionItem title="Agent 1 — Trade Signal: Full Workflow" icon="📈" badge="HITL">

**State (the robot's clipboard):**

| Field | Description |
|-------|-------------|
| `asset_symbol` | The stock or asset ticker (e.g., 005930.KS for Samsung) |
| `market` | Which market: KR, US, crypto, forex |
| `ohlcv_data` | Open/High/Low/Close/Volume price data |
| `indicators` | Calculated technical indicators (RSI, MACD, etc.) |
| `market_regime` | Current market condition (bull, bear, sideways) |
| `signal_direction` | BUY, SELL, or HOLD |
| `confidence_pct` | How confident the model is (0-100%) |
| `price_target` | Predicted target price |
| `stop_loss` | Recommended stop-loss level |
| `compliance_result` | PASS or FAIL from compliance check |
| `order_id` | Staged order identifier |
| `user_decision` | What the human decided (approve/reject/modify) |
| `retry_count` | Number of retries attempted |

**Workflow (step-by-step):**

| Step | Node | What Happens |
|------|------|-------------|
| 1 | `fetch_market_data` | Pulls OHLCV data via kis-trading MCP |
| 2 | `calculate_indicators` | Runs TA-Lib: RSI, MACD, Bollinger, ATR, VWAP, Ichimoku |
| 3 | `evaluate_regime` | Determines if market is bullish, bearish, or sideways |
| 4 | `score_signal` | XGBoost scores the signal; Claude explains reasoning |
| 5 | `compliance_check` | Validates against FSC rules via compliance-check MCP |
| 6 | `notify` | If compliance PASS, sends signal to user |
| 7 | **`await_user_decision`** | **HITL node** — waits for human approval (30-minute timeout). If timeout expires, the order is cancelled. |
| 8 | `stage_order` | If approved, stages the order via kis-trading MCP |
| 9 | `log_outcome` | Writes immutable audit record |

<InfoBox type="warning" title="Non-negotiable HITL">
  The await_user_decision node is non-negotiable. Even if the user enables "Auto Mode" in settings, this step still requires explicit human confirmation for every trade. This is an FSC regulatory requirement.
</InfoBox>

</AccordionItem>

  <AccordionItem title="Agent 2 — AI Advisor Chatbot: Full Workflow" icon="💬" badge="Streaming">

**State (the robot's clipboard):**

| Field | Description |
|-------|-------------|
| `user_id` | Who is asking the question |
| `question_text` | The raw question from the user |
| `question_type` | Classified type (market, portfolio, general, etc.) |
| `retrieval_plan` | Which sources to query for context |
| `market_context` | Data from market-context MCP |
| `portfolio_context` | Data from portfolio-intelligence MCP |
| `news_context` | Data from pgvector news embeddings |
| `response_text` | The generated answer |
| `review_passed` | Whether self-review passed (true/false) |
| `review_failures` | List of what failed in self-review |
| `citations` | Source references for the answer |
| `retry_count` | Number of retries attempted |

**Workflow (step-by-step):**

| Step | Node | What Happens |
|------|------|-------------|
| 1 | `classify_question` | Determines what kind of question the user asked |
| 2 | `plan_retrieval` | Decides which data sources to query |
| 3 | `fetch_context` | **PARALLEL** fetch: market-context MCP + portfolio-intelligence MCP + pgvector news search |
| 4 | `load_cached_context` | Loads stable knowledge from CAG (Context-Augmented Generation) — pre-loaded facts that never need retrieval |
| 5 | `generate_response` | Claude generates a streaming answer using all gathered context |
| 6 | `self_review` | FSC compliance self-check: Does the response include a disclaimer? Are citations present? Does it avoid making guarantees? |
| 7 | `stream_to_user` | Sends the response to the user in real time |
| 8 | `append_citations` | Attaches source references at the end |
| 9 | `log_interaction` | Writes immutable audit record |

<InfoBox type="tip" title="What is CAG?">
  CAG (Context-Augmented Generation) is like giving the robot a cheat sheet stapled to its desk. Instead of searching the library every time someone asks "What is KOSPI?", the answer is already loaded in memory. This saves time and money. See the Intelligence Layer page for full CAG details.
</InfoBox>

</AccordionItem>

  <AccordionItem title="Agent 3 — Risk Guard: Full Workflow" icon="🛡️" badge="HITL + AML">

**State (the robot's clipboard):**

| Field | Description |
|-------|-------------|
| `transaction_id` | The transaction being evaluated |
| `user_id` | The user who initiated the transaction |
| `amount` | Transaction amount |
| `risk_score` | Anomaly score from Isolation Forest (0.0 to 1.0) |
| `aml_flags` | Anti-money laundering flags triggered |
| `sanctions_hit` | Whether the user matches a sanctions list |
| `account_status` | Current account state (active, frozen, etc.) |
| `str_draft_text` | Suspicious Transaction Report draft text |
| `officer_decision` | Compliance officer's decision |
| `resolution_timestamp` | When the case was resolved |

**Workflow (step-by-step):**

| Step | Node | What Happens |
|------|------|-------------|
| 1 | `score_anomaly` | Isolation Forest scores the transaction 0.0 (normal) to 1.0 (suspicious) |
| 2 | `aml_check` | Checks KoFIU thresholds: transactions at or above KRW 10M flagged; sanctions list cross-check |
| 3a | `approve` | If score is below 0.3 and no AML flags: transaction approved automatically |
| 3b | `enhanced_monitoring` | If score is 0.3 to 0.7: transaction proceeds but is flagged for enhanced monitoring |
| 3c | `freeze_and_alert` | If score is above 0.7 OR any AML flag: account frozen, compliance team alerted |
| 4 | `generate_str_draft` | Claude drafts a Suspicious Transaction Report (STR) with all evidence summarized |
| 5 | **`await_compliance_review`** | **HITL node** — compliance officer reviews the case (72-hour timeout; if no response, auto-escalates to senior compliance) |
| 6 | `execute_decision` | Carries out the officer's decision (unfreeze, file STR, escalate) |
| 7 | `log_outcome` | Writes immutable audit record |

<InfoBox type="note" title="Three risk lanes">
  Think of it like a security checkpoint at an airport. Green lane (below 0.3) — walk right through. Yellow lane (0.3 to 0.7) — extra screening but you can still proceed. Red lane (above 0.7 or AML flag) — you're stopped until a security officer clears you.
</InfoBox>

</AccordionItem>

  <AccordionItem title="Agent 4 — Market Analysis: Full Workflow" icon="📊" badge="Multi-market">

**State (the robot's clipboard):**

| Field | Description |
|-------|-------------|
| `assets_data` | Raw market data across all tracked markets |
| `indicators_map` | Technical indicators computed per asset |
| `news_context` | Relevant news articles retrieved via RAG |
| `analysis_text` | The generated market analysis |
| `quality_passed` | Whether quality check passed |
| `quality_failures` | List of quality issues found |
| `retry_count` | Number of retries attempted |
| `published_at` | Timestamp of publication |
| `notification_triggers` | Events that trigger user notifications |

**Workflow (step-by-step):**

| Step | Node | What Happens |
|------|------|-------------|
| 1 | `fetch_all_market_data` | **PARALLEL** fetch across 8 markets: KR, US, JP, CN, HK, VN, crypto, forex |
| 2 | `run_indicators` | TA-Lib computes RSI, MACD, Bollinger Bands, ATR, VWAP, and Ichimoku Cloud for each asset |
| 3 | `fetch_relevant_news` | pgvector RAG retrieves news articles relevant to current market conditions |
| 4 | `generate_analysis` | Claude writes the analysis in 3 languages: Korean (KO), English (EN), and Chinese (CN) |
| 5 | `quality_check` | Validates completeness, accuracy of cited data, and disclaimer presence |
| 6 | `cache_and_publish` | Stores the analysis in Redis with a 15-minute TTL, then publishes |
| 7 | `trigger_notifications` | Sends alerts to users whose watchlists match significant movements |

<InfoBox type="fun" title="Morning newspaper for robots">
  This agent is like a financial journalist who wakes up every morning, reads 8 different newspapers from around the world, does the math, and writes a summary. Except this journalist writes in 3 languages simultaneously and never sleeps.
</InfoBox>

</AccordionItem>

  <AccordionItem title="Agent 5 — Portfolio Optimizer: Full Workflow" icon="⚖️" badge="HITL + Loop">

**State (the robot's clipboard):**

| Field | Description |
|-------|-------------|
| `user_id` | The portfolio owner |
| `holdings` | Current list of assets held |
| `current_weights` | Percentage allocation per asset |

## LangGraph Capabilities

<InfoBox type="info" title="The operating system for agents">
  If agents are the robots, LangGraph is their operating system. It defines how each robot follows its instruction manual (workflow), when it needs to ask a human (HITL), and how it remembers where it left off if the power goes out (checkpointing).
</InfoBox>

### Core Capabilities

| Capability | Description | Why It Matters |
|------------|-------------|----------------|
| **Typed Shared State** | Every agent has a strongly-typed state object (like a clipboard) that is passed between nodes | Prevents bugs caused by missing or wrong data types — if a field is supposed to be a number, it cannot accidentally be text |
| **HITL Nodes** | Special pause points where the workflow stops and waits for a human | Required by FSC regulations for all financial actions; prevents autonomous money movement |
| **Conditional Edges** | Branching logic: "if risk score above 0.7, go to freeze; otherwise, go to approve" | Enables smart routing without hardcoding every possible path |
| **Parallel Execution** | Multiple nodes can run at the same time (e.g., fetching data from 3 sources simultaneously) | Dramatically reduces latency for multi-source data gathering |
| **Checkpointer Persistence** | Full agent state saved to Redis at every node transition | If the server crashes mid-workflow, the agent resumes exactly where it stopped — no lost progress, no repeated work |
| **Streaming Output** | Agents can stream partial results to users in real time (e.g., chatbot typing effect) | Users see responses as they are generated, not after a long wait |

### HITL Timeout Summary

Every HITL node has a timeout — if the human does not respond in time, the system takes a safe default action.

| Agent | HITL Node | Timeout | Default Action on Timeout |
|-------|-----------|---------|--------------------------|
| **Trade Signal** | `await_user_decision` | 30 minutes | Order cancelled |
| **Risk Guard** | `await_compliance_review` | 72 hours | Auto-escalate to senior compliance |
| **Portfolio Optimizer** | `await_decision` | 7 days | Recommendation archived |
| **CUI Agent** | `await_confirmation` | 5 minutes | Action cancelled |

<InfoBox type="tip" title="Safe defaults protect everyone">
  Every timeout results in a safe action — cancel, escalate, or archive. No timeout ever results in "do it anyway." This is a core safety principle: when in doubt, do nothing and tell a human.
</InfoBox>

## Prompt & Context Engineering

<InfoBox type="fun" title="Teaching robots to read and talk">
  Prompt engineering is like writing really good instructions for a new employee. Context engineering is like making sure the employee has the right documents on their desk before they start working. Together, they determine how well our AI agents understand questions and give useful answers.
</InfoBox>

### Prompt Engineering Techniques

<InfoBox type="info" title="6 techniques for crafting AI instructions">
  These are the core strategies we use to write prompts (instructions) for Claude and other LLMs. Each technique makes the AI's output more reliable, more compliant, and more useful.
</InfoBox>

<Accordion>
  <AccordionItem title="1. Role-Based System Prompts" icon="🎭" badge="Foundation">

Every agent receives a system prompt that defines its identity, scope, and constraints. This is the first thing the AI reads, and it shapes everything that follows.

**Example pattern:**
- "You are the Risk Guard agent for BARO. Your only job is to evaluate transactions for AML compliance. You MUST include a disclaimer in every output. You CANNOT approve transactions — you can only recommend actions to a human compliance officer."

<InfoBox type="tip" title="Why this works">
  Giving the AI a clear role is like giving a new employee a job description on day one. Without it, the AI might try to be helpful in ways that are actually dangerous — like approving a transaction it should only be flagging.
</InfoBox>

</AccordionItem>

  <AccordionItem title="2. Structured Output Formatting" icon="📋" badge="Reliability">

Prompts specify the exact output format expected — JSON schemas, required fields, and validation rules. This ensures downstream systems can parse AI outputs reliably.

**Example pattern:**
- "Respond with a JSON object containing: signal_direction (BUY/SELL/HOLD), confidence_pct (0-100), reasoning (string, max 500 characters), and disclaimer (string, must include FSC required text)."

</AccordionItem>

  <AccordionItem title="3. Chain-of-Thought Prompting" icon="🔗" badge="Reasoning">

For complex decisions, prompts ask the AI to show its work step by step before giving a final answer. This makes reasoning transparent and auditable.

**Example pattern:**
- "Before providing your risk assessment, walk through these steps: (1) List all anomaly indicators present, (2) Compare against KoFIU thresholds, (3) Check sanctions list results, (4) State your conclusion with a risk score and justification."

</AccordionItem>

  <AccordionItem title="4. Few-Shot Examples" icon="📝" badge="Accuracy">

Prompts include 2-5 example inputs and expected outputs so the AI understands the exact format and quality level expected.

**Example pattern:**
- "Here are 3 examples of correctly formatted market analysis summaries: [Example 1: KR market bullish...] [Example 2: US market mixed...] [Example 3: Crypto bearish...] Now write a summary for today's data following the same format."

</AccordionItem>

  <AccordionItem title="5. Guardrail Prompting" icon="🚧" badge="Safety">

Explicit instructions about what the AI must NOT do. These negative constraints are as important as positive instructions.

**Example pattern:**
- "NEVER provide specific buy/sell recommendations without the compliance disclaimer. NEVER claim to guarantee returns. NEVER reference data you were not given in the context window. If you are unsure, say so explicitly."

</AccordionItem>

  <AccordionItem title="6. Multi-Language Templating" icon="🌐" badge="Localization">

Prompts include language-specific templates for Korean, English, and Chinese output. Financial terminology has precise translations that must be consistent.

**Example pattern:**
- "Generate the analysis in Korean. Use the following financial terms consistently: 매수 (buy), 매도 (sell), 보유 (hold), 손절 (stop-loss), 목표가 (target price)."

</AccordionItem>
</Accordion>

### Context Engineering Techniques

<InfoBox type="info" title="8 techniques for giving AI the right information">
  Context engineering is about what information the AI has access to when it generates a response. Even the smartest AI gives bad answers if it is working from bad or missing information. These 8 techniques ensure every agent has exactly the right context — no more, no less.
</InfoBox>

<Accordion>
  <AccordionItem title="1. RAG (Retrieval-Augmented Generation)" icon="🔍" badge="Dynamic">

The workhorse of context engineering. When an agent needs information, it searches the vector database (pgvector) using BGE-M3 embeddings and retrieves the most relevant chunks.

**How it works:** User question → embed with BGE-M3 → search pgvector → retrieve top-K chunks → feed to Claude as context → generate answer.

**Used by:** AI Advisor, Document Analysis, Market Analysis, News Sentiment

</AccordionItem>

  <AccordionItem title="2. CAG (Context-Augmented Generation)" icon="📖" badge="Cached">

Stable, rarely-changing knowledge is pre-loaded into the context window at startup instead of being retrieved every time. This is faster and cheaper than RAG for information that does not change often.

**Examples of CAG content:**
- FSC regulatory requirements and disclaimer text
- Korean financial terminology glossary
- Company ticker-to-name mappings
- Standard technical indicator definitions

**How it differs from RAG:** RAG goes to the library every time. CAG is like a cheat sheet stapled to the desk — always there, no lookup needed.

**Used by:** AI Advisor, Trade Signal, Risk Guard, Market Analysis

See the [Intelligence Layer](/ai-platform/intelligence-layer) page for the full CAG specification.

</AccordionItem>

  <AccordionItem title="3. Hybrid Search (Semantic + Keyword)" icon="🔀" badge="Precision">

Combines pgvector semantic search with BM25 keyword search. Semantic search understands meaning ("stock going up" matches "bullish equity trend"). Keyword search catches exact terms that semantic search might miss ("005930.KS").

**Used by:** Document Analysis, News Sentiment

</AccordionItem>

  <AccordionItem title="4. Reranking" icon="🏆" badge="Quality">

After initial retrieval returns 20 candidates, a Cohere reranker scores them by relevance and keeps only the top 5. This dramatically improves the quality of context fed to Claude.

**Used by:** Document Analysis

</AccordionItem>

  <AccordionItem title="5. Parallel Context Fetching" icon="⚡" badge="Speed">

When an agent needs data from multiple sources, it fetches them all simultaneously instead of one at a time. For example, the AI Advisor fetches market data, portfolio data, and news context in parallel.

**Used by:** AI Advisor, Market Analysis, News Sentiment

</AccordionItem>

  <AccordionItem title="6. Permission-Filtered Context" icon="🔒" badge="Security">

Before any retrieved context is shown to Claude, it passes through an access control filter. Chunks from documents the user is not authorized to see are removed. Claude never sees unauthorized data.

**Used by:** Document Analysis

</AccordionItem>

  <AccordionItem title="7. Time-Windowed Context" icon="⏰" badge="Freshness">

Market data and news context is time-windowed to ensure the AI only works with recent, relevant information. Old data is filtered out based on configurable TTLs (e.g., market analysis uses the last 15 minutes of cached data).

**Used by:** Market Analysis, Trade Signal, News Sentiment

</AccordionItem>

  <AccordionItem title="8. Entity Resolution" icon="🏷️" badge="Accuracy">

Natural language references are resolved to canonical system identifiers before any context lookup. "Samsung" becomes "005930.KS", "Apple" becomes "AAPL", "Tesla" becomes "TSLA". This ensures the right data is fetched every time.

**Used by:** CUI Agent, AI Advisor, Trade Signal

</AccordionItem>
</Accordion>
