  <Tab label="CAG Deep Dive">
    ### Claude Prompt Caching (CAG)

    CAG uses **Claude Prompt Caching** with `cache_control` breakpoints. Cached content has a **5-minute TTL** and is reused across requests, slashing costs dramatically.

    <InfoBox type="warning" title="The Golden Rule: Byte-for-Byte Identical">
      For prompt caching to work, the cached portion of the prompt must be **byte-for-byte identical** across requests. Even a single character difference breaks the cache and you pay full price. This is why the 7 prompt layers are ordered from most stable to least stable.
    </InfoBox>

    ### The 7 Prompt Layers

    Ordered from most stable (top, always cached) to least stable (bottom, never cached):

    <StepGuide>
      <Step number={1} title="System Prompt" status="complete">
        Core identity and behavior rules for each agent. Changes only on deployment. **Always cached.**
      </Step>
      <Step number={2} title="Stable Knowledge (CAG)" status="complete">
        FSC regulations, market structure rules, glossary. Updated weekly at most. **Always cached.**
      </Step>
      <Step number={3} title="Session Context" status="complete">
        User profile, risk tolerance, portfolio summary. Stable within a session. **Usually cached.**
      </Step>
      <Step number={4} title="Dynamic RAG Results" status="active">
        Retrieved document chunks relevant to the current query. Changes per request. **Not cached.**
      </Step>
      <Step number={5} title="Live Market Data" status="active">
        Real-time prices, volumes, order book snapshots. Changes every second. **Not cached.**
      </Step>
      <Step number={6} title="Conversation History" status="active">
        Recent messages in the current chat session. Grows with each turn. **Not cached.**
      </Step>
      <Step number={7} title="User Query" status="active">
        The actual question or command from the user. Unique every time. **Never cached.**
      </Step>
    </StepGuide>

    ### Pipeline Stages in Detail

    <Accordion>
      <AccordionItem title="Stage 1 — Ingestion: 7 Input Modalities" icon="📥" badge="Multi-format">
        The pipeline accepts 7 distinct input types, each with a specialized converter that produces clean, embeddable text.

        <InfoBox type="fun" title="Think of it this way">
          The pipeline is like a multilingual translator who can read books, look at pictures, listen to recordings, and watch videos -- then summarize everything in one common language that Claude can understand.
        </InfoBox>

        | # | Modality | Source Examples | Converter Stack | Output Format |
        |---|---|---|---|---|
        | 1 | PDF Research Reports | KIS/Mirae broker notes, FSC filings | PyMuPDF + Google Doc AI OCR + pdfplumber | Structured text + table JSON |
        | 2 | Charts and Financial Images | Broker charts, TradingView screenshots | LLaVA-1.6 (Railway GPU) or GPT-4o Vision fallback | 150-300 word domain captions |
        | 3 | Audio (Earnings Calls) | Samsung/SK IR, Bank of Korea pressers | Whisper large-v3 + Naver Clova Speech | Timestamped transcript with speaker diarization |
        | 4 | Video (Investor Presentations) | Webcasts, factory tours | FFmpeg to Whisper + frames to LLaVA | Interleaved multimodal transcript |
        | 5 | Financial Tables | KIS exports, DART tables | openpyxl + pandas + PyMuPDF | Label-preserving tabular text |
        | 6 | HTML and Web Content | Naver Finance, KRX announcements | BeautifulSoup + Playwright | Clean article text |
        | 7 | User Interaction Data | Chatbot conversations, trade signals | PostgreSQL export | JSONL instruction-tuning format, PIPA anonymization nightly |
      </AccordionItem>

      <AccordionItem title="Stage 2 — Conversion and OCR" icon="🔄" badge="0.85 confidence threshold">
        Raw files are converted to clean text. For scanned documents and images, OCR is applied with a **confidence threshold of 0.85**. Any document where OCR confidence falls below 0.85 is flagged for **human review** before entering the pipeline.

        <InfoBox type="note" title="Why 0.85?">
          Financial documents contain numbers where a single wrong digit can change the meaning entirely. A confidence score below 0.85 means the OCR is not sure enough about the text -- and in finance, "pretty sure" is not good enough.
        </InfoBox>
      </AccordionItem>

      <AccordionItem title="Stage 3 — Semantic Chunking" icon="✂️" badge="200-400 tokens">
        Documents are split into semantically meaningful chunks rather than arbitrary fixed-size windows.

        **Chunking rules:**
        - **Prose text:** 200-400 tokens per chunk
        - **Tables:** Never split -- always kept as a complete unit
        - **Overlap:** 50 tokens between adjacent chunks to preserve context at boundaries
        - **Metadata attached to every chunk:**

        | Metadata Field | Purpose |
        |---|---|
        | `source_doc_id` | Trace back to the original document |
        | `page_number` | Locate the chunk within the document |
        | `section_heading` | Understand the chunk's context |
        | `modality` | Know what type of content this is (text, table, caption) |
        | `language` | Route to correct language processing |
        | `access_level` | Enforce data access permissions |

        <InfoBox type="fun" title="Think of it this way">
          Chunking is like cutting a long article into index cards. Each card has enough text to make sense on its own, plus a little overlap with the card before it so you don't lose the thread. Tables are never cut in half -- you always keep the whole table on one card.
        </InfoBox>
      </AccordionItem>

      <AccordionItem title="Stage 4 — Embedding" icon="📐" badge="BGE-M3 primary">
        Each chunk is converted into a high-dimensional vector that captures its semantic meaning.

        | Model | Dimensions | Cost | Role |
        |---|---|---|---|
        | **BGE-M3** | 1024 | $0 (local) | **Primary** -- multilingual, runs on our infrastructure |
        | text-embedding-3-small | 1536 | Per-token API cost | **Fallback** -- OpenAI hosted, used if BGE-M3 is down |
        | CLIP ViT-L/14 | 512 | $0 (local) | **Phase 3** -- image embeddings for chart similarity |
        | ImageBind | 1024 | $0 (local) | **Phase 3** -- cross-modal (text to image to audio) |

        <InfoBox type="fun" title="Think of it this way">
          Embedding is like assigning GPS coordinates to every index card. Cards about similar topics end up with similar coordinates, so when someone asks a question, we can find the "nearest" cards by checking which coordinates are closest.
        </InfoBox>
      </AccordionItem>

      <AccordionItem title="Stages 5-6 — Hybrid Retrieval" icon="🔍" badge="3 strategies">
        When a user query arrives, three retrieval strategies work together to find the most relevant chunks.

        **pgvector schema:**

        | Column | Type | Purpose |
        |---|---|---|
        | `chunk_id` | UUID | Unique chunk identifier |
        | `doc_id` | UUID | Parent document reference |
        | `text_content` | TEXT | The actual chunk text |
        | `embedding` | VECTOR(1024) | BGE-M3 dense vector |
        | `modality` | ENUM | Content type (text, table, caption, transcript) |
        | `page_number` | INT | Source page |
        | `section_heading` | TEXT | Section context |
        | `language` | ENUM | KO, EN, or CN |
        | `access_level` | ENUM | Public, user, admin |

        **Three retrieval strategies combined:**

        | Strategy | Weight | How It Works |
        |---|---|---|
        | Dense Vector Search | 50% | Cosine similarity against BGE-M3 embeddings -- finds semantically similar content |
        | BM25 Keyword Search | 30% | Traditional keyword matching -- catches exact terms that vectors might miss |
        | Metadata Filtering | Applied first | Filters by language, access level, modality, and date range before any scoring |

        Results from dense and BM25 search are combined using **Reciprocal Rank Fusion (RRF)**, which merges ranked lists by giving higher scores to chunks that appear near the top in multiple strategies.

        <InfoBox type="fun" title="Think of it this way">
          Imagine asking three librarians to each find the 10 best books for your question. One librarian picks books by meaning (vector search), another picks by exact keywords (BM25), and the third filters out books you are not allowed to read (metadata). Then they compare notes -- books that all three librarians agree on go to the top of the stack.
        </InfoBox>
      </AccordionItem>

      <AccordionItem title="Stage 7 — Re-ranking" icon="🏆" badge="Largest quality jump">
        After hybrid retrieval returns a candidate set, a dedicated reranker re-scores every chunk against the original query for fine-grained relevance.

        | Reranker | Cost | Role |
        |---|---|---|
        | **Cohere Rerank API** | $1 per 1,000 calls | **Primary** -- highest quality semantic reranking |
        | BGE Reranker v2 | $0 (local) | **Fallback** -- used if Cohere is unavailable |

        **This stage produces the single largest quality improvement in the entire pipeline.** Hybrid retrieval casts a wide net (high recall), but reranking precisely orders the results (high precision). Without reranking, the top chunks often include near-misses that waste context window tokens.

        <InfoBox type="note" title="Why reranking matters most">
          The difference between "pretty relevant" and "exactly what was asked" is enormous when you only have 2,000 tokens of context budget. Reranking ensures those 2,000 tokens contain the best possible information.
        </InfoBox>
      </AccordionItem>

      <AccordionItem title="Stage 8 — Context Assembly" icon="📦" badge="~3,600 tokens typical">
        The final stage assembles all context into a structured prompt that fits within Claude's token budget.

        **Token budget breakdown (typical request):**

        | Component | Tokens | Source |
        |---|---|---|
        | System prompt | ~500 | CAG (cached) |
        | User query | ~100 | Direct input |
        | Retrieved chunks | ~2,000 | RAG pipeline output |
        | Live market data | ~300 | Real-time feeds |
        | User profile context | ~200 | Session state |
        | Conversation history | ~500 | Recent messages |
        | **Total** | **~3,600** | **Combined** |

        The assembler respects strict ordering: cached content first (to maintain cache hits), then dynamic content, then the user query last. This ordering is critical for prompt caching to work -- remember the byte-for-byte identical rule.

        <InfoBox type="fun" title="Think of it this way">
          Context assembly is like packing a suitcase with a strict weight limit. You have to fit the essentials (system prompt, user question), the best research (top-ranked chunks), and some useful extras (market data, chat history) -- all within about 3,600 tokens. The packing order matters too: heavy stable items go at the bottom (cached), lighter changing items on top.
        </InfoBox>
      </AccordionItem>
    </Accordion>

    - Audio tone analysis demonstrably adds signal value beyond transcription alone
    - Text-only retrieval quality becomes a measurable bottleneck (retrieval precision drops below target)

  <Tab label="Local LLM">
    ### Local LLM Training Strategy

    The Local LLM is not built on Day 1. Instead, data collection starts from Day 1, and the first fine-tuned model is trained once enough high-quality data accumulates.

    <InfoBox type="fun" title="Think of it this way">
      Training a local model is like raising a student. First, you let them observe the expert teacher (Claude) handling real questions. You carefully record every lesson. After enough observations, the student can handle routine questions on their own -- freeing the expert teacher for the hard problems.
    </InfoBox>

    ### Data Collection from Day 1

    Every interaction is logged to a `training_events` table for future fine-tuning. All data collection is **PIPA compliant** (Korea's Personal Information Protection Act).

    | PIPA Requirement | Implementation |
    |---|---|
    | User consent | Explicit opt-in during onboarding |
    | Training opt-in | Separate toggle in user settings |
    | Deletion rights | User can request full data deletion at any time |
    | Anonymization | Nightly batch job strips all PII from training data |

    ### 6 Data Types Collected

    <Accordion>
      <AccordionItem title="1. Chatbot Interactions" icon="💬" badge="Highest volume">
        User questions and Claude's responses from the AI Advisor. Captures natural Korean financial language, user intent patterns, and effective response structures.
      </AccordionItem>

      <AccordionItem title="2. Trade Signal Events" icon="📊" badge="Decision data">
        Signal generation inputs, outputs, and whether the user acted on the signal. Teaches the model how to interpret technical indicators and generate actionable signals.
      </AccordionItem>

      <AccordionItem title="3. Document Analyses" icon="📄" badge="Domain structure">
        How Claude parsed and summarized DART filings, broker reports, and FSC disclosures. Teaches document structure understanding specific to Korean financial documents.
      </AccordionItem>

      <AccordionItem title="4. Market Analysis Sessions" icon="📈" badge="Reasoning chains">
        Full analysis reasoning chains including which data points were considered, how they were weighted, and the final conclusion.
      </AccordionItem>

      <AccordionItem title="5. Compliance Decisions" icon="🛡️" badge="Critical accuracy">
        Risk Guard outputs including flagged transactions, reasoning, and outcomes. Requires the highest accuracy bar before the local model can take over.
      </AccordionItem>

      <AccordionItem title="6. Expert Annotations" icon="👨‍💼" badge="Gold standard">
        Human expert corrections and ratings of AI outputs. These are the highest-value training examples because they directly encode domain expertise.
      </AccordionItem>
    </Accordion>

    </InfoBox>

    ### QLoRA Fine-Tuning

    | Parameter | Value |
    |---|---|
    | Hardware | 1x NVIDIA A100 (80GB) |
    | Training time | 12-24 hours per run |
    | Cost per run | $30-80 |
    | Quality vs. full fine-tune | Approximately 90% |
    | Memory savings | 4-bit quantization reduces VRAM from 60GB+ to under 24GB |

    <InfoBox type="fun" title="Think of it this way">
      Full fine-tuning is like rebuilding an entire car engine to make it faster. QLoRA is like adding a turbocharger -- you get 90% of the speed improvement at a fraction of the cost, and you can swap it out easily if you want to try a different configuration.
    </InfoBox>

  <Tab label="Dual LLM Routing">
    ### Dual LLM Routing Strategy

    Not every query needs Claude. The routing layer decides whether a query goes to the expensive cloud LLM (Claude) or the free local LLM, based on 6 dimensions.

    <InfoBox type="fun" title="Think of it this way">
      Imagine a hospital triage system. Critical patients go to the specialist (Claude). Routine checkups go to the general practitioner (Local LLM). The triage nurse (router) decides who goes where based on how complex, urgent, and high-stakes the situation is.
    </InfoBox>

    ### 6 Routing Dimensions

    | Dimension | Routes to Local LLM | Routes to Claude |
    |---|---|---|
    | Query Complexity | Simple lookups, FAQs, template responses | Multi-step reasoning, novel questions |
    | Domain Familiarity | Well-covered in training data | Edge cases, unusual instruments |
    | Stakes | Informational queries, general education | Compliance, trade execution, regulatory |
    | Output Format | Standard templates, short answers | Complex analysis, detailed explanations |
    | Latency Requirement | Latency-sensitive (notifications) | Quality-sensitive (advisory) |
    | Language | Korean (strongest local model language) | English/Chinese or mixed-language |

    ### Traffic Split Over Time

    <StepGuide>
      <Step number={1} title="Phase 1: Cloud Only" status="complete">
        **0% local traffic.** All queries go to Claude. Data collection is active. No local model exists yet.
      </Step>
      <Step number={2} title="Phase 2 Early: First Fine-Tune" status="active">
        **20-30% local traffic.** First EXAONE fine-tune handles simple Korean FAQs and template responses. Rule-based router.
      </Step>
      <Step number={3} title="Phase 2 Mature: Expanding Coverage" status="active">
        **50-60% local traffic.** Multiple fine-tuning iterations. Local model handles standard advisory, basic signals, and document summaries.
      </Step>
      <Step number={4} title="Phase 3: Confidence-Based Routing" status="pending">
        **70-80% local traffic.** ML-based router with confidence scoring. Local model handles most Korean financial queries.
      </Step>
      <Step number={5} title="Long-Term Steady State" status="pending">
        **85-90% local traffic.** Continuous training loop. Claude reserved for novel queries, compliance, and complex multi-step reasoning.
      </Step>
    </StepGuide>

    ### Confidence-Based Fallback

    Starting in Phase 3, the router uses a **confidence threshold of 0.75**. If the local model's confidence score falls below 0.75 on any query, it automatically routes to Claude instead.

    <InfoBox type="warning" title="Safety net, not a gamble">
      The confidence fallback ensures users never receive a low-quality response just to save money. If the local model is not confident, Claude takes over transparently. The user never knows the difference.
    </InfoBox>
