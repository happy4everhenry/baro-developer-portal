The operational backbone of BARO AI — a 7-stage audit lifecycle that records every action by every actor (human or AI), safe execution rails that prevent AI from going rogue, and One Cockpit to monitor it all in real time. Operations and audit infrastructure is being built alongside Phase 1 agent development.

## The 7-Stage Interaction Lifecycle

This is the centerpiece of BARO's operational integrity. Every single interaction — whether initiated by a human user or an AI agent — passes through these stages.

<StepGuide>
  <Step number={1} title="Identity Check" icon="🔐">

**JWT + Session + MFA.** The actor is always known before any action proceeds. Whether it's a human user logging in or an AI agent initiating a workflow, identity is verified first. No anonymous actions exist in the system.

</Step>

  <Step number={2} title="History & Context" icon="📋">

**Previous actions, open flags, compliance status, portfolio state.** The system loads the full context of the actor — what they've done before, what flags are active on their account, their current compliance standing, and the state of their portfolio. This prevents actions that conflict with existing conditions.

</Step>

  <Step number={3} title="Summary & Flags" icon="🚩">

**Active flags surfaced, notifications acknowledged.** Before any action is drafted, all active flags (compliance holds, risk warnings, pending reviews) are presented. The actor must acknowledge them. Nothing is hidden.

</Step>

  <Step number={4} title="Draft" icon="📝">

**Action composed but not submitted.** The intended action is drafted with full detail. A compliance pre-check runs automatically. Even **abandoned drafts are recorded** — if someone starts composing a suspicious trade and backs out, we know about it.

</Step>

  <Step number={5} title="Approval" icon="✅">

**Single or Dual approval depending on risk level.**
- **Single approval:** User confirms the action themselves (standard transactions)
- **Dual approval:** Compliance officer + user confirmation required (transactions exceeding KRW 50M)

</Step>

  <Step number={6} title="Submit" icon="🚀">

**Order sent to KIS API, database updated.** The action is executed. Every parameter of the submission is logged — the exact payload sent to KIS, timestamps, response codes, and the resulting database state change.

</Step>

  <Step number={7} title="Immutable Record" icon="🔒">

**All events consolidated. Written once, never modified.** The complete interaction is sealed into an append-only audit record. Every stage's data is bundled together. This record cannot be edited or deleted — only new records can be appended.

</Step>
</StepGuide>

### 5-Minute Audit Capability

From a single `interaction_id`, anyone with the right permissions can reconstruct the full story in under 5 minutes:

| Question | Answer Source |
|----------|-------------|
| Who initiated? | Stage 1 — Identity Check |
| What verification passed? | Stage 1 — JWT, session, MFA records |
| What flags were active? | Stage 3 — Flag snapshot at interaction time |
| What was shown in draft? | Stage 4 — Full draft content preserved |
| Who approved? | Stage 5 — Approval actor + timestamp |
| What was submitted? | Stage 6 — Exact API payload logged |
| What was the outcome? | Stage 7 — Consolidated immutable record |

<InfoBox type="info" title="No developer required">
  The 5-minute audit is designed for compliance officers, not engineers. The One Cockpit interface lets authorized users pull up any interaction by ID and walk through every stage visually. No SQL queries, no log files, no asking the dev team.
</InfoBox>

### AI Agents in the Audit Trail

AI agents are **first-class actors** — they go through the same 7 stages as humans. The stages they touch depend on the action:

| Agent | Action | Stages Used | Human Confirm? | Audit Log Detail |
|-------|--------|------------|----------------|------------------|
| CUI | Navigates screen | 1, 4, 6, 7 | No | agent=CUI, action=navigate_to |
| CUI | Pre-fills order | 1, 4, 6, 7 | No | parameters pre-filled, draft_event |
| CUI | Submits staged order | **All 7** | Stage 5 HITL mandatory | Full interaction_record + approval_event |
| Trade Signal | Generates signal | 1, 4, 7 | No (advisory only) | signal generated |
| Risk Guard | Flags transaction | 1, 2, 3, 4, 5, 7 | Compliance officer at Stage 5 | flag_registry entry |
| Risk Guard | Freezes account | **All 7** | Compliance officer must approve | Cannot be AI alone |
| Portfolio Optimizer | Recommends allocation | 1, 4, 7 | No (recommendation only) | recommendation_record |
| Portfolio Optimizer | Executes rebalance | **All 7** | User must approve full plan | Full record for each leg |
| Feature Request | Generates spec | 1, 6, 7 | No (document generation) | spec_document_record |
| Agent 11 | Quarantines agent | 1, 5, 6, 7 | Automated, human notified immediately | quarantine_event + alert |

<InfoBox type="warning" title="HITL = Human-in-the-Loop">
  Any action that touches money or restricts account access **always** requires human confirmation. An AI agent can draft, recommend, and flag — but it cannot execute financial actions or freeze accounts on its own.
</InfoBox>

### Safe AI Execution Rails

Three categories of safety rails ensure AI agents operate within strict boundaries. Each rail addresses a different type of risk.

<Accordion>
  <AccordionItem title="Rail 1: UI Action Safety" icon="🖥️" defaultOpen={true}>

Controls how AI agents interact with the user interface. Prevents confusing or harmful screen changes.

| Rule | Purpose |
|------|---------|
| **Narration before navigation** | AI must announce what it's about to do before changing screens |
| **Undo always available** | Every AI-initiated screen change can be reversed by the user |
| **Screen changes audited** | Every navigation event by an AI agent is recorded in the audit trail |

#### Detailed Implementation

| Rail | Implementation | Enforced By |
|------|---------------|-------------|
| Narration before navigation | CUI Agent narrates what it will do BEFORE doing it (e.g., "Opening Samsung order form") | `generate_narration` node runs before `execute_actions` node |
| Undo navigation | Any AI navigation can be undone with "go back." AI-initiated nav tagged in app state. | App state tracks `navigation_source`: AI or user |
| Screen change audit | Every App Action API call writes to `ai_action_log` with state before and after | App Action API middleware |
| No UI actions outside session | CUI cannot act after session timeout. Session expiry terminates pending AI actions. | JWT validation on every App Action API call |

<InfoBox type="fun" title="Think of it like a GPS co-pilot">
  Imagine a friend who grabs the steering wheel without telling you where they are turning. Terrifying, right? Narration before navigation means the AI always says "turning left toward Samsung's order page" before it touches anything. And if you disagree, you can always say "go back" — like pressing undo on your GPS route.
</InfoBox>

<InfoBox type="tip" title="Why narration matters">
  If the CUI agent suddenly jumps to a different screen without explanation, the user might think the app is malfunctioning. Narration builds trust: "I'm taking you to Samsung Electronics' detail page to show the chart you asked about."
</InfoBox>

</AccordionItem>

  <AccordionItem title="Rail 2: Database Action Safety" icon="💾">

Controls how AI agents read from and write to the database. Prevents data leaks and unauthorized modifications.

| Rule | Purpose |
|------|---------|
| **RLS enforcement** | Row-Level Security ensures agents only see data they're authorized to access |
| **Scoped reads/writes per agent** | Each agent has a defined scope — it can only read/write specific tables and columns |
| **No direct audit table writes** | AI agents cannot write to audit tables directly; only the audit system can |
| **Redis namespace isolation** | Each agent's cache is isolated in its own Redis namespace — no cross-agent data leakage |

**Read Scope:** RLS-scoped per agent. CUI reads only the current user's portfolio. Enforced at the Supabase RLS level — the agent never sees rows belonging to other users.

**Write Scope:** Each agent writes only to allowed tables. CUI is limited to `ai_action_log` and `draft_events` only. No agent can write outside its designated tables.

**No Direct Audit Writes:** A dedicated audit service writes all immutable records. Agents fire events; the audit service writes. This separation prevents self-falsification — an agent cannot alter its own audit trail.

**DB Logging:** Every AI write is logged with `agent_id`, `table`, `operation`, `row_id`, `old_value`, and `new_value` via a PostgreSQL trigger. Nothing is written silently.

**Redis Namespace Isolation:** Each agent writes only to its own key namespace. For example, CUI uses `cui:session:*` and `cui:action:*`. No cross-agent writes are possible.

</AccordionItem>

  <AccordionItem title="Rail 3: Money Action Safety" icon="💰">

The most critical rail. Controls how financial transactions flow through the system.

**The golden rule: AI stages, human confirms, system executes, audit records.**

This is a non-negotiable principle. No configuration, no override, no admin setting can bypass this flow for financial transactions.

| Phase | Actor | Action |
|-------|-------|--------|
| **Stage** | AI Agent | Drafts the transaction, runs pre-checks, calculates parameters |
| **Confirm** | Human | Reviews and explicitly approves the transaction |
| **Execute** | System | Sends the order to KIS API (not the AI, not the human) |
| **Record** | Audit System | Writes the immutable record with all details |

#### Money Action Authority Matrix

Every type of financial action has explicit rules for what the AI can do, what requires human confirmation, and when dual approval kicks in.

| Money Action | AI Authority | Human Required | Dual Approval | Timeout |
|---|---|---|---|---|
| Small stock order | Stage only, cannot execute | User tap confirm (HITL) | N/A | 30-min, order cancelled |
| Large order (above KRW 50M) | Stage only | User + Compliance Officer | Yes, KRW 50M threshold | 60-min, both required |
| Portfolio rebalancing | Recommendation only | User approves full plan | All rebalancing | 7-day, each leg separate |
| Conditional/automated order | Stage only, cannot arm | User explicitly enables trigger | All conditional orders | 30-day re-confirmation |
| Account funding/withdrawal | Cannot initiate, advisory only | User initiates manually | N/A | N/A |
| AML transaction freeze | Can recommend with evidence | Compliance Officer authorizes | All freezes | 72-hour, escalate if no response |

<InfoBox type="fun" title="Think of it like a restaurant kitchen">
  The AI is the sous chef who prepares ingredients and plates the dish (stages). The human is the head chef who tastes and approves before it leaves the kitchen (confirms). The waiter (system) delivers it to the table (executes). And the receipt (audit) records the entire order. The sous chef never walks a dish to the table alone — no matter how perfectly it is plated.
</InfoBox>

<InfoBox type="warning" title="The Golden Rule is non-negotiable">
  "AI stages. Human confirms. System executes. Audit records." This cannot be shortcut by context, urgency, or any technical mechanism. Even if an AI agent has 99.99% confidence in a trade recommendation, the human must still confirm. This is a regulatory requirement (FSC) and a core BARO design principle.
</InfoBox>

</AccordionItem>

## DAGS Continuous Loop

The operational lifecycle follows a continuous **Discover-Assess-Govern-Secure-Audit** loop. This is not a one-time process — it runs perpetually to ensure governance keeps pace with system evolution.

<StepGuide>
  <Step number={1} title="Discover" icon="🔍">

Identify all AI assets, agents, data flows, and model deployments. Maintain a living inventory of what exists in the system. New agents, new models, new data sources — nothing enters the platform without being discovered and cataloged first.

</Step>

  <Step number={2} title="Assess" icon="📊">

Evaluate risk, compliance posture, and ethical alignment for every discovered asset. Each agent is assessed against the 5 governance pillars, regulatory requirements, and the threat model. Assessment is continuous, not one-time.

</Step>

  <Step number={3} title="Govern" icon="⚖️">

Apply governance controls based on assessment results. This includes updating system prompts, adjusting authorization boundaries, modifying kill switch thresholds, and ensuring the Legal Rules Engine reflects current regulations.

</Step>

  <Step number={4} title="Secure" icon="🔒">

Implement and validate security controls for all assessed risks. Run daily posture scans, monthly red team exercises, and continuous monitoring through Agent 11 (Security Monitor) and Agent 12 (Control Tower).

</Step>

  <Step number={5} title="Audit" icon="📋">

Record everything through the 7-stage audit lifecycle. Generate compliance reports, conduct periodic reviews, and feed findings back into the Discover phase — completing the loop. Every audit finding triggers a new cycle.

</Step>
</StepGuide>

<InfoBox type="fun" title="Think of it like a health checkup loop">
  DAGS is like going to the doctor regularly. You discover what is in your body (Discover), check if anything is wrong (Assess), follow the doctor's rules (Govern), take your medicine (Secure), and keep a medical record (Audit). Then you go back next month and do it again. You never stop checking.
</InfoBox>

### DAGS Step-by-Step Execution Detail

<Accordion>
  <AccordionItem title="DISCOVER — Threat and Asset Inventory" icon="🔍" defaultOpen={true}>

**What:** Identify all AI assets, agents, data flows, model deployments, shadow AI usage, and dependency vulnerabilities. Maintain a living registry of threats.

**Who/What runs it:** Agent 11 (Security Monitor) + Security Lead

**Output:** Threat registry, shadow AI inventory, dependency vulnerability list

**Schedule:** Hourly automated feeds, daily full scan

</AccordionItem>

  <AccordionItem title="ASSESS — Risk Scoring and Prioritization" icon="📊">

**What:** Score every discovered item for impact and likelihood. Prioritize remediation. Track deadlines for open items.

**Who/What runs it:** Agent 11 + Agent 12 (Control Tower) + Security Lead

**Output:** Impact assessment per item, prioritized remediation backlog, deadline tracker

**Schedule:** Daily automated assessment, weekly human review

</AccordionItem>

  <AccordionItem title="GOVERN — Controls and Compliance" icon="⚖️">

**What:** Apply governance controls. Update the HITL queue. Run bias audits. Produce governance scorecards.

**Who/What runs it:** Compliance Officer + Agent 12

**Output:** Governance scorecard, HITL queue updates, bias audit results

**Schedule:** Continuous automated checks, daily compliance review, monthly full governance audit

<InfoBox type="info" title="Human review required">
  The GOVERN step always requires human sign-off. Agent 12 prepares the scorecard, but a Compliance Officer must review and approve it before controls are applied. This is enforced in the loop execution record.
</InfoBox>

</AccordionItem>

  <AccordionItem title="SECURE — Posture Validation" icon="🔒">

**What:** Run posture scans, execute canary tests, conduct penetration tests, rotate secrets.

**Who/What runs it:** Agent 11 + Security Lead

**Output:** Posture scan results, canary test results, pen test findings, secrets rotation log

**Schedule:** Daily posture scan, weekly canary probes, monthly penetration test

</AccordionItem>

  <AccordionItem title="AUDIT — Record and Report" icon="📋">

**What:** Verify audit chain integrity. Produce compliance packages for FSC. Identify gaps and feed back into DISCOVER.

**Who/What runs it:** Agent 12 + Compliance Officer

**Output:** Chain verification results, FSC compliance package, gap list for next cycle

**Schedule:** Daily automated checks, monthly compliance sign-off, quarterly external audit preparation

<InfoBox type="info" title="Human review required">
  Like GOVERN, the AUDIT step requires human sign-off. Agent 12 produces the compliance package, but a Compliance Officer must review and sign before submission to the FSC.
</InfoBox>

</AccordionItem>
</Accordion>

### Loop Execution Record

Every DAGS cycle is tracked in the `loop_execution_log` table — an append-only, tamper-evident record that ensures no step is skipped, delayed, or falsified.

| Column | Type | Description |
|--------|------|-------------|
| `cycle_id` | `UUID` | Unique identifier for this DAGS cycle |
| `step` | `VARCHAR` | DISCOVER, ASSESS, GOVERN, SECURE, or AUDIT |
| `executed_by` | `VARCHAR` | Agent ID or human role that ran this step |
| `started_at` | `TIMESTAMPTZ` | When this step began |
| `completed_at` | `TIMESTAMPTZ` | When this step finished |
| `findings_count` | `INTEGER` | Number of findings produced by this step |
| `items_resolved` | `INTEGER` | Number of items resolved during this step |
| `output_document_ids` | `UUID[]` | References to output documents (scorecards, reports, etc.) |
| `status` | `ENUM` | `COMPLETED`, `PARTIAL`, or `FAILED` |
| `human_reviewed` | `BOOLEAN` | Required for GOVERN and AUDIT steps |
| `next_scheduled_run` | `TIMESTAMPTZ` | When the next execution of this step is expected |

<InfoBox type="fun" title="Think of it like a pilot's pre-flight checklist">
  Pilots do not skip checklist items even if they have flown the same plane a thousand times. The loop execution record is the checklist — every step must be completed, checked off, and signed. If a step says PARTIAL or FAILED, the system flags it immediately, like a co-pilot saying "we missed item 7, we cannot take off yet."
</InfoBox>

## Alerting Severity Framework

All alerts across the 9 monitoring layers are classified into 4 severity levels. Each level has a defined auto-response, notification chain, and SLA.

| Severity | Definition | Auto Response | Human Notify | SLA |
|---|---|---|---|---|
| **P0 Critical** | Platform down or money at risk | Classic Mode activated, freeze orders, quarantine agents | PagerDuty on-call + CTO, Slack #incidents, SMS to security | Ack: 5 min, Mitigate: 30 min, Resolve: 2 hours |
| **P1 High** | Significant degradation or compliance risk | Route to fallback services, increase logging level | PagerDuty on-call, Slack #incidents | Ack: 15 min, Mitigate: 1 hour, Resolve: 4 hours |
| **P2 Medium** | Degraded performance or elevated risk | Auto-remediation (restart services, clear cache) | Slack #alerts, email to on-call | Ack: 30 min, Resolve: 24 hours |
| **P3 Low** | Anomaly within normal range, needs investigation | Log and queue for review | Slack #monitoring, daily digest | Review: next business day |

<InfoBox type="fun" title="Think of it like a hospital triage">
  P0 is a heart attack — everyone drops what they are doing and runs. P1 is a broken arm — urgent, but the heart attack patient goes first. P2 is a bad cold — the nurse handles it on schedule. P3 is a routine checkup — it can wait until tomorrow. The triage system ensures the right response at the right speed.
</InfoBox>

<InfoBox type="warning" title="P0 auto-response is immediate">
  When a P0 fires, the system does not wait for a human. Classic Mode activates automatically, pending orders freeze, and compromised agents quarantine themselves. Human notification happens simultaneously — but the system protects users first, then explains why.
</InfoBox>

## Chaos Playbooks

Seven pre-defined playbooks for the most critical failure scenarios. Each playbook has been rehearsed and is triggered automatically or semi-automatically based on the alerting framework.

<InfoBox type="fun" title="Think of it like fire drills">
  Schools practice fire drills so that when a real fire happens, nobody panics — everyone knows exactly where to go. Chaos playbooks work the same way. We have already decided what to do for every major disaster, written it down, practiced it, and wired the system to start executing the plan automatically.
</InfoBox>

<Accordion>
  <AccordionItem title="PB-001: Claude API Outage" icon="🧠" defaultOpen={true}>

**Trigger:** Claude API unreachable or latency exceeds SLA for 60+ seconds.

**Auto Response:**
- Route 100% of traffic to local LLM fallback
- Downgrade AI features to local model capabilities
- Alert CTO and engineering on-call

**Recovery:** Monitor Claude status. Canary test before restoring traffic. Gradual ramp-up (10% then 50% then 100%).

</AccordionItem>

  <AccordionItem title="PB-002: KIS API Outage" icon="📈">

**Trigger:** KIS API unreachable or order submission failure rate exceeds 5%.

**Auto Response:**
- Halt all new order submissions
- Freeze pending orders in staged state
- Display maintenance banner to users
- Contact KIS support

**Escalation:** If orders were mid-execution when outage occurred, notify FSC compliance team. User funds may be in transit.

**Recovery:** Confirm KIS API health. Reconcile all pending orders. Resume in order of submission time.

</AccordionItem>

  <AccordionItem title="PB-003: Database Overload" icon="🗄️">

**Trigger:** Query latency p95 exceeds 500ms or connection pool exceeds 95%.

**Auto Response:**
- Route reads to read replica
- Shed non-critical write operations
- Queue AI training and analytics writes

**Recovery:** Identify and terminate long-running queries. Scale connection pool if needed. Restore normal routing once p95 drops below 100ms.

</AccordionItem>

  <AccordionItem title="PB-004: Coordinated Prompt Injection" icon="🛡️">

**Trigger:** Agent 11 detects injection attempt spike across multiple agents simultaneously.

**Auto Response:**
- Quarantine affected agents immediately
- Activate Classic Mode for all users
- Block source IPs at edge
- Increase logging to maximum verbosity

**Recovery:** Review injection logs. Identify attack vector. Patch prompt defenses. Run canary tests against new attack patterns. Reactivate agents one-by-one with monitoring.

</AccordionItem>

  <AccordionItem title="PB-005: AML Transaction Freeze" icon="🚩">

**Trigger:** Risk Guard agent flags a transaction for anti-money laundering review.

**Procedure:**
1. Risk Guard raises flag with evidence
2. Compliance Officer reviews within 72 hours
3. If confirmed: file KoFIU Suspicious Transaction Report (STR) within 3 business days
4. FSC notification required for all confirmed AML events

**Escalation:** If no Compliance Officer response within 72 hours, auto-escalate to CTO and external compliance counsel.

</AccordionItem>

  <AccordionItem title="PB-006: Data Breach" icon="🔓">

**Trigger:** Agent 11 detects unauthorized data access or exfiltration attempt.

**Auto Response:**
- Revoke all active sessions immediately
- Isolate affected systems from network
- Increase logging to maximum on all systems
- Begin forensic investigation

**Regulatory Requirements:**
- PIPC (Personal Information Protection Commission) notification within 72 hours
- Affected users notified within 72 hours
- Preserve all evidence for investigation

</AccordionItem>

  <AccordionItem title="PB-007: KRX Circuit Breaker" icon="⚡">

**Trigger:** Korea Exchange (KRX) activates market-wide or stock-specific circuit breaker.

**Auto Response:**
- Halt all order submissions automatically
- Display circuit breaker notice to all active users
- Monitor KRX announcements for resumption

**Recovery:** Resume order acceptance only after KRX confirms trading has reopened. Notify users with staged orders that they must re-confirm.

</AccordionItem>
</Accordion>
