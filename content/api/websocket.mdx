<InfoBox type="info" title="What is a WebSocket?">
Imagine a REST API is like sending a letter and waiting for a reply. A WebSocket is like a phone call -- once you connect, both sides can talk to each other instantly, back and forth, without hanging up. This is how BARO gets stock prices that update in real time!
</InfoBox>

## Connection

<ApiEndpoint method="WS" path="ws://host:8000/ws/stream" description="Bidirectional real-time streaming">

iOS clients connect to this endpoint to receive real-time trade ticks and order book updates. The server acts as a relay between iOS clients and the upstream KIS WebSocket feed.

| Environment | URL | Notes |
|------------|-----|-------|
| iOS Simulator | `ws://127.0.0.1:8000/ws/stream` | Localhost via simulator |
| Physical Device | `ws://192.168.68.57:8000/ws/stream` | Local network IP (update for your network) |
| Production | `wss://api.baro.app/ws/stream` | TLS-secured (future) |

</ApiEndpoint>

## Connection Lifecycle

<StepGuide>
<Step number={1} title="Client connects">
Client opens WebSocket to `/ws/stream`. Server accepts and assigns a `client_id` (UUID prefix).
</Step>
<Step number={2} title="Server sends status">
Server sends an initial `status` message with the KIS connection state.
</Step>
<Step number={3} title="Client subscribes">
Client sends `subscribe` with stock codes and venue. Server confirms with `subscribed` message.
</Step>
<Step number={4} title="Data streams in">
Server streams `trade` and `orderbook` updates as they happen. Multiple updates per second during active trading.
</Step>
<Step number={5} title="Keep-alive pings">
Client sends `ping` every 30 seconds. Server responds with `pong`.
</Step>
<Step number={6} title="Cleanup on disconnect">
When the client disconnects, the server automatically cleans up subscriptions.
</Step>
</StepGuide>

## Reconnection Strategy

<InfoBox type="tip" title="Automatic reconnection">
Both the iOS client and the backend use exponential backoff -- they wait a little longer each time a reconnection fails, up to a maximum of 60 seconds. Once reconnected, all subscriptions are automatically restored.
</InfoBox>

<Accordion>
<AccordionItem title="iOS Client (WebSocketManager)" icon="📱" badge="Client" defaultOpen={true}>

- **Initial delay:** 1.0 seconds
- **Backoff multiplier:** 2x (exponential)
- **Max delay:** 60 seconds
- **Reset:** On successful reconnection
- **Disconnect delay:** 5 seconds grace period to avoid churn during navigation
- **Re-subscribe:** All (code, venue) pairs re-sent on reconnect

```swift
private func attemptReconnect() {
    guard reconnectTask == nil else { return }

    reconnectTask = Task { [weak self] in
        guard let self else { return }

        while !Task.isCancelled {
            print("[WS] Reconnecting in \(self.reconnectDelay)s...")
            try? await Task.sleep(for: .seconds(self.reconnectDelay))
            guard !Task.isCancelled else { return }

            // Clean up old connection
            self.webSocketTask?.cancel(with: .abnormalClosure, reason: nil)
            self.webSocketTask = nil

            // Attempt reconnection
            let request = URLRequest(url: Self.wsURL)
            self.webSocketTask = self.session.webSocketTask(with: request)
            self.webSocketTask?.resume()

            // Verify with ping
            do {
                try await withCheckedThrowingContinuation { continuation in
                    self.webSocketTask?.sendPing { error in
                        if let error { continuation.resume(throwing: error) }
                        else { continuation.resume() }
                    }
                }
                // Success -- reset backoff and re-subscribe
                self.isConnected = true
                self.reconnectDelay = 1.0
                self.resubscribeAll()
                return
            } catch {
                // Failed -- increase backoff
                self.reconnectDelay = min(
                    self.reconnectDelay * 2,
                    self.maxReconnectDelay
                )
            }
        }
    }
}
```

</AccordionItem>
<AccordionItem title="Backend to KIS" icon="⚙️" badge="Server">

- **Initial delay:** 1.0 seconds
- **Backoff multiplier:** 2x (exponential)
- **Max delay:** 60 seconds
- **Reset:** On successful connection
- **Approval key:** Cleared and re-requested on reconnect failure
- **Re-subscribe:** All tracked tr_id:tr_key pairs re-sent

</AccordionItem>
</Accordion>

## Limits & Constraints

| Constraint | Value | Notes |
|-----------|-------|-------|
| Max KIS subscriptions | `~40` | Per upstream WebSocket session. Shared across all iOS clients via reference counting. |
| Trading hours (KRX) | `09:00--15:30 KST` | Trade ticks only arrive during market hours. NXT has extended hours. |
| Supported markets | Domestic only | KOSPI/KOSDAQ real-time via KIS WS. Overseas stocks not supported via WebSocket. |
| Ping interval | `30 seconds` | iOS client sends keep-alive ping. KIS upstream uses PINGPONG heartbeat. |
| Disconnect grace period | `5 seconds` | iOS delays disconnect to avoid churn during view navigation. |
| Reconnect max backoff | `60 seconds` | Both iOS client and backend cap exponential backoff at 60s. |

## Client to Server Messages

<InfoBox type="note" title="Message format">
All messages are JSON objects. The `action` field tells the server what you want to do.
</InfoBox>

<ApiEndpoint method="POST" path="subscribe" description="Subscribe to stock codes for real-time updates">

Subscribe to receive real-time trade ticks and order book updates for the specified stock codes. The server reference-counts subscriptions -- if another client is already watching the same code, no additional KIS subscription is created.

**Schema**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `action` | `string` | Required | Must be `"subscribe"` |
| `codes` | `string[]` | Required | Array of stock codes (digits only). e.g. `["005930", "000660"]` |
| `venue` | `string` | Optional | Market venue: `"combined"`, `"krx"`, `"nxt"`. Default: `"combined"` |

**Example**

```json
{
  "action": "subscribe",
  "codes": ["005930", "000660", "035420"],
  "venue": "combined"
}
```

**Server Response**

```json
{
  "type": "subscribed",
  "codes": ["005930", "000660", "035420"],
  "venue": "combined"
}
```

</ApiEndpoint>

<ApiEndpoint method="POST" path="unsubscribe" description="Unsubscribe from stock codes">

Stop receiving updates for the specified stock codes. If no other clients are watching the same (code, venue) pair, the upstream KIS subscription is released.

**Schema**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `action` | `string` | Required | Must be `"unsubscribe"` |
| `codes` | `string[]` | Required | Array of stock codes to unsubscribe from |
| `venue` | `string` | Optional | Market venue. Default: `"combined"` |

**Example**

```json
{
  "action": "unsubscribe",
  "codes": ["005930"],
  "venue": "combined"
}
```

**Server Response**

```json
{
  "type": "unsubscribed",
  "codes": ["005930"],
  "venue": "combined"
}
```

</ApiEndpoint>

<ApiEndpoint method="POST" path="ping" description="Keep-alive heartbeat">

Send periodic ping messages to keep the connection alive. The iOS client sends a ping every 30 seconds.

**Client sends**

```json
{"action": "ping"}
```

**Server responds**

```json
{"type": "pong"}
```

</ApiEndpoint>

<ApiEndpoint method="POST" path="status" description="Request streaming status">

Request the current streaming subsystem status. Returns the same data as `GET /ws/status` but over the WebSocket channel.

**Client sends**

```json
{"action": "status"}
```

**Server responds**

```json
{
  "type": "status",
  "kis_connected": true,
  "active_subscriptions": 3,
  "connected_clients": 2,
  "watched_codes": ["005930@combined"]
}
```

</ApiEndpoint>

## Server to Client Messages

<ApiEndpoint method="WS" path="trade" description="Real-time trade tick">

Received whenever a trade executes for a subscribed stock. Contains the latest price, change, and volume data. Typically arrives multiple times per second during active trading.

**Schema**

| Field | Type | Description |
|-------|------|-------------|
| `type` | `string` | Always `"trade"` |
| `code` | `string` | Stock code (e.g. `"005930"`) |
| `venue` | `string` | Source venue: `"combined"`, `"krx"`, `"nxt"` |
| `price` | `integer` | Execution price |
| `change` | `integer` | Change from previous close (negative for decline) |
| `change_percent` | `float` | Percentage change |
| `trade_time` | `string` | Execution time in `HHMMSS` format (KST) |
| `volume` | `integer` | Trade volume (number of shares in this execution) |
| `cumulative_volume` | `integer` | Cumulative volume for the trading day |

**Example**

```json
{
  "type": "trade",
  "code": "005930",
  "venue": "combined",
  "price": 72300,
  "change": -200,
  "change_percent": -0.28,
  "trade_time": "103025",
  "volume": 150,
  "cumulative_volume": 12345678
}
```

</ApiEndpoint>

<ApiEndpoint method="WS" path="orderbook" description="Real-time order book update">

Received when the order book changes for a subscribed stock. Contains up to 10 levels of ask (sell) and bid (buy) prices with quantities.

**Schema**

| Field | Type | Description |
|-------|------|-------------|
| `type` | `string` | Always `"orderbook"` |
| `code` | `string` | Stock code |
| `venue` | `string` | Source venue |
| `asks` | `object[]` | Ask levels: `[{price, quantity}, ...]` (up to 10) |
| `bids` | `object[]` | Bid levels: `[{price, quantity}, ...]` (up to 10) |
| `total_ask_quantity` | `integer` | Total ask-side volume |
| `total_bid_quantity` | `integer` | Total bid-side volume |

**Example**

```json
{
  "type": "orderbook",
  "code": "005930",
  "venue": "combined",
  "asks": [
    { "price": 72400, "quantity": 1500 },
    { "price": 72500, "quantity": 2100 },
    { "price": 72600, "quantity": 890 }
  ],
  "bids": [
    { "price": 72300, "quantity": 2300 },
    { "price": 72200, "quantity": 1800 },
    { "price": 72100, "quantity": 4500 }
  ],
  "total_ask_quantity": 45000,
  "total_bid_quantity": 52000
}
```

</ApiEndpoint>

<ApiEndpoint method="DELETE" path="error" description="Server error message">

Sent when the server encounters an error processing a client message. Common causes: invalid JSON, missing required fields, unknown action. These errors do not close the connection.

**Schema**

| Field | Type | Description |
|-------|------|-------------|
| `type` | `string` | Always `"error"` |
| `message` | `string` | Human-readable error description |

**Error Examples**

```json
{"type": "error", "message": "codes must be a non-empty list"}
{"type": "error", "message": "No valid stock codes provided"}
{"type": "error", "message": "Invalid JSON"}
{"type": "error", "message": "Unknown action: foo"}
```

</ApiEndpoint>

## iOS Client Integration

### WebSocketManager Usage

The `WebSocketManager` is an `@Observable` singleton providing two ways to consume real-time data:

<Accordion>
<AccordionItem title="Pattern 1: Observe latestPrices Dictionary" icon="👀" badge="SwiftUI" defaultOpen={true}>

```swift
// In a SwiftUI View -- triggers re-render when any price updates
let wsMgr = WebSocketManager.shared

var body: some View {
    if let update = wsMgr.latestPrices["005930"] {
        Text("\(update.price.formattedWon)")
            .contentTransition(.numericText())
            .animation(.easeInOut(duration: 0.3), value: update.price)
    }
}

// Lifecycle
.onAppear {
    wsMgr.connect()
    wsMgr.subscribe(codes: ["005930"], venue: .combined)
}
.onDisappear {
    wsMgr.unsubscribe(codes: ["005930"], venue: .combined)
    wsMgr.disconnectAfterDelay(seconds: 5.0)
}
```

</AccordionItem>
<AccordionItem title="Pattern 2: Per-Code AsyncStream" icon="🔄" badge="ViewModel">

```swift
// In a ViewModel -- async consumption of individual stock updates
func startLiveUpdates() {
    WebSocketManager.shared.connect()
    WebSocketManager.shared.subscribe(codes: [stock.code])

    liveUpdateTask = Task {
        for await update in WebSocketManager.shared.updates(for: stock.code) {
            livePrice = update.price
            liveChange = update.change
            liveChangePercent = update.changePercent
        }
    }
}

func stopLiveUpdates() {
    liveUpdateTask?.cancel()
    WebSocketManager.shared.unsubscribe(codes: [stock.code])
    WebSocketManager.shared.disconnectAfterDelay()
}
```

</AccordionItem>
</Accordion>

## KIS Upstream Protocol

The backend maintains a persistent WebSocket connection to KIS at `ws://ops.koreainvestment.com:21000`.

### Venue to TR_ID Mapping

| Venue | Trade TR_ID | Orderbook TR_ID | REST FID Code |
|-------|-------------|-----------------|---------------|
| `krx` (Korea Exchange) | `H0STCNT0` | `H0STASP0` | `J` |
| `nxt` (NextRade Korea) | `H0NXCNT0` | `H0NXASP0` | `NX` |
| `combined` (Unified) | `H0UNCNT0` | `H0UNASP0` | `UN` |

### KIS Wire Format

<InfoBox type="note" title="Not JSON!">
KIS real-time data uses a pipe-and-caret delimited binary format, not JSON. The backend parses this and converts it to clean JSON for iOS clients.
</InfoBox>

```python
# KIS real-time message format:
# encrypt_flag | tr_id | count | body
# Body fields are caret (^) delimited

# Example raw message:
# 0|H0STCNT0|001|005930^103025^72300^2^-200^-0.28^...

# Parsed as:
encrypt_flag = "0"          # 0 = unencrypted
tr_id        = "H0STCNT0"  # KRX trade tick
count        = "001"        # number of records
body         = "005930^103025^72300^2^-200^-0.28^..."
```

Special messages:

```python
# KIS heartbeat -- must echo back immediately
if msg.startswith("PINGPONG"):
    await ws.send("PINGPONG")

# JSON subscription confirmations
# {"header": {"tr_id": "H0STCNT0", "tr_key": "005930"}, "body": {"rt_cd": "0"}}
```

<Accordion>
<AccordionItem title="H0STCNT0 Trade Tick Fields" icon="📊" badge="Parse Map">

| Index | KIS Field Name | Mapped To | Description |
|-------|----------------|-----------|-------------|
| `0` | MKSC_SHRN_ISCD | `code` | Stock code (6 digits) |
| `1` | STCK_CNTG_HOUR | `trade_time` | Execution time (HHMMSS) |
| `2` | STCK_PRPR | `price` | Current/execution price |
| `3` | PRDY_VRSS_SIGN | `sign` | Change sign: 1-3 positive, 4-5 negative |
| `4` | PRDY_VRSS | `change` | Absolute change from prev close |
| `5` | PRDY_CTRT | `change_percent` | Percentage change |
| `7` | STCK_OPRC | `open` | Opening price |
| `8` | STCK_HGPR | `high` | High price |
| `9` | STCK_LWPR | `low` | Low price |
| `12` | CNTG_VOL | `volume` | Trade volume (this tick) |
| `13` | ACML_VOL | `cumulative_volume` | Cumulative day volume |
| `14` | ACML_TR_PBMN | `cumulative_value` | Cumulative trading value |

</AccordionItem>
<AccordionItem title="H0STASP0 Orderbook Fields" icon="📖" badge="Parse Map">

| Index Range | Field | Description |
|-------------|-------|-------------|
| `0` | Stock code | 6-digit stock code |
| `3--12` | askp1--askp10 | Ask prices (10 levels, lowest first) |
| `13--22` | bidp1--bidp10 | Bid prices (10 levels, highest first) |
| `23--32` | askp_rsqn1--10 | Ask quantities per level |
| `33--42` | bidp_rsqn1--10 | Bid quantities per level |
| `43` | total_askp_rsqn | Total ask volume |
| `44` | total_bidp_rsqn | Total bid volume |

</AccordionItem>
<AccordionItem title="KIS Subscription Message Format" icon="📨" badge="JSON">

```json
{
  "header": {
    "approval_key": "your-approval-key-here",
    "custtype": "P",
    "tr_type": "1",
    "content-type": "utf-8"
  },
  "body": {
    "input": {
      "tr_id": "H0STCNT0",
      "tr_key": "005930"
    }
  }
}
```

`tr_type`: `"1"` = subscribe, `"2"` = unsubscribe. The `approval_key` is obtained via `POST /oauth2/Approval` (see [Authentication](/api/authentication)).
